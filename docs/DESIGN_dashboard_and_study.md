# 설계 — 관리 대시보드 + 공부 Q&A 담기(/study)

목표 두 가지를 기존 파이프라인(**저작은 대화, 배선은 스크립트**) 위에 얹는다.
새 사실을 지어내지 않는 원칙(원문 사실만)과 세션 간 지속성(git 추적)을 그대로 지킨다.

> 상태: **설계 확정 대기**. 아래 "열린 결정"에 답을 주면 구현에 들어간다.

---

## A) 관리 대시보드 — 한 페이지에서 검토 상태 관리

### A-1. 무엇을 보여주나
`assets/dashboard.html` 한 장에서 세 묶음을 표로 본다.

- **문서**(assets/*.html): 이름 · 검토상태 · 난이도 분포(쉬움/보통/어려움) · 린트 ⚠수 · 얼버무림 후보수 · 검은박스 · 최종 갱신 · [열기]
- **일차**(days/N): 일차 · 붙은 문서 · 문제 유무 · 일차덱 유무 · 상태 · [열기]
- **PPT 덱**(out/*.pptx): 이름 · 소스 문서 · 슬라이드수 · 빌드 여부

각 행의 **검토상태 칩**: `미검토(회색) · 검토중(주황) · ✅검토완료(초록)`. 팔레트는 참고 디자인(Oracle Red/Slate) 재사용.

### A-2. 상태 저장 — `review_status.json` (git 추적, 사람 소유)
```json
{
  "docs": {
    "guide_62_68.html": {
      "status": "reviewed",              // 사람이 관리(대시보드가 덮어쓰지 않음)
      "reviewed_at": "2026-07-01",
      "note": "LLM 리뷰 반영"
    }
  },
  "days": {
    "71": { "status": "published", "note": "" }
  }
}
```
- **사람 소유 필드**(status/reviewed_at/note)는 이 파일에만 있고 재생성 때 보존.
- **자동 필드**(난이도 분포·린트수·얼버무림수·검은박스·슬라이드수·갱신시각)는 대시보드 생성 때 **매번 재계산**해서 병합 → 표에 표시(파일에 저장 안 함).

### A-3. 생성기 — `build_dashboard.py`
```
python build_dashboard.py            # assets/·days/·out/ 스캔 → assets/dashboard.html
```
내부 동작:
1. `lint_authoring.lint()` 로 문서별 난이도 분포·보강권장수.
2. `lint_vagueness.scan` 으로 얼버무림 후보수.
3. `sync_and_verify._defined_classes` + SVG 스캔으로 검은박스.
4. `review_status.json`(사람 상태) 병합.
5. 정적 HTML 렌더(자바스크립트 최소, 링크는 상대경로).

### A-4. 상태 토글 — `review.py`(작은 CLI)
```
python review.py mark guide_62_68.html --status reviewed --note "확인함"
python review.py list                       # 현재 상태 요약
```
→ `review_status.json` 갱신 후 `build_dashboard.py` 자동 호출. 대화로도 "guide 검토완료로 표시해줘" 하면 내가 이 명령을 돌린다.

### A-5. 파이프라인 연결
- `make_day.py` 끝에 **"5) 대시보드 갱신"** 단계 추가(항상 최신).
- 덱(out/*.pptx)은 생성물이라 git 제외 → 대시보드는 **빌드 여부/슬라이드수**만 표시하고, 없으면 "생성 명령" 안내. 링크는 그 세션에서 빌드됐을 때만 유효.

## B) 공부 Q&A → 담기 선택 (/study 스킬)

### B-1. 흐름
```
/study <모르는 부분 질문>
  1. 내가 설명 — 가능하면 assets/ 문서를 근거로. 문서에 없으면 '일반 지식'으로 명시.
  2. 끝에 선택지(AskUserQuestion):
        [문서에 담기] [PPT에 담기] [둘 다] [설명만 보기]
  3. 담기 선택 시:
        · 문서: 알맞은 assets/<topic>.html 에 개념 섹션 추가(AUTHORING.md 난이도 매칭)
                → sync_and_verify(재주입·검증) + 린트 + 대시보드 갱신
        · PPT : 해당 문서 덱 재생성 또는 days/<N> 일차덱에 슬라이드 추가 → build
        · 새로 담긴 개념은 review_status = '미검토'로 등록(당신이 나중에 검토완료 토글)
```

### B-2. 어느 문서로 가나
주제로 자동 추정: `sql_tuning`(실행계획·인덱스·CF) / `buffer_cache`(DBWR·checkpoint·redo) /
`rman_recovery`(백업·복구) / `guide_62_68`(SGA·latch·ITL 등 내부구조). 애매하면 물어본다.

### B-3. 안전장치 (원문 사실만 원칙 유지)
- 답변이 **문서 근거**면 그대로 구조화해 담는다.
- 답변이 **일반 지식(문서 밖)**이면: 담기 전에 "이건 노트 밖 일반 지식입니다 — 담을까요?"로
  한 번 더 확인하고, 본문에 근거 구분 표시. **불확실하면 '확인 필요'로 남기고 지어내지 않는다.**
- 담긴 개념은 얼버무림·난이도 린트를 통과해야 최종 반영(벽글·얼버무림이면 대화로 보강).

### B-4. 구현 형태
- `.claude/skills/study/SKILL.md` — 위 흐름을 담은 스킬(세션에서 `/study`로 호출).
- 실제 저작은 내가 대화로(스킬이 규약·체크리스트를 로드), 배선은 기존 스크립트 재사용.

## 세션 간 지속성 (이미 확보)
- `assets/`·`days/`·`review_status.json`·`dashboard.html`·`.claude/skills/` 모두 **git 추적**.
- 생성물(`out/`·`*.pptx`)만 제외 → 필요 시 세션에서 빌드/다운로드.
- 하루 작업 끝에 커밋/푸시하면 다음 세션에 그대로 복원.

## 자동화 수준(선택)
- **반자동(권장)**: 노트를 리포에 올리면 트리거된 세션이 `make_day`→PR 생성, 당신은 PR 검토.
  저작 판단은 세션의 LLM이 담당하므로 "매번 손으로 안 돌려도 됨". 완전 무인은 원문 원칙상 비권장.

## 열린 결정 (구현 전 확인)
1. **대시보드 상태 축**: `미검토/검토중/검토완료` 3단계면 충분? (예: `보류`·`재작업` 추가?)
2. **/study 담기 기본값**: 기본을 '설명만'으로 두고 담기는 명시 선택? (실수 반영 방지)
3. **문서 밖 일반 지식**: 담기 허용? 허용 시 본문에 '노트 밖' 배지로 구분하는 방식 OK?
4. **대시보드 위치**: `assets/dashboard.html`(허브 옆) vs 리포 루트 `dashboard.html`?
5. **구현 순서**: 대시보드 먼저 → /study 나중? (둘 다 선택하셨고 '설계만'이라 순서만 확정하면 착수)

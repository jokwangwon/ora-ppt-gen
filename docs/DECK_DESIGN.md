# 덱 설계 원칙 — 노트 덤프가 아니라 '이해시키는' 발표 자료

세 갈래 조사(Assertion-Evidence · Mayer 멀티미디어 학습원리 · 시각설계)를 종합해,
하루치 수업을 **통째로 이해한 뒤** 만드는 깔끔한 덱의 규약. `build_ppt.js`·`layout.js`·`preview.js`에 반영.

## 왜 바꿨나 — 기존 덱의 문제
- 제목이 **주제어**("row migration")라 요지를 안 말함
- 문서의 다이어그램이 슬라이드엔 **텍스트 요약**으로만 들어감
- 한 슬라이드에 블록을 몰아넣고 자동 페이지네이션 → **한 블록씩 쪼개져 여백만 큼**
- 상세 설명이 **화면에 그대로** = 발표자가 읽을 게 화면에 다 있음(청중은 읽으며 못 들음)

## 0. 축 잡기 — 저작 전에 먼저 (가장 중요)
덱을 쓰기 *전에* 원문(노트·문서)을 훑어 **그날의 진짜 축(중심 주제)** 을 먼저 찾는다.
축을 잘못 잡으면 내용이 다 맞아도 발표가 겉돈다 (72일차: '측정→비교→선택'으로 잡았으나
실제 축은 **'실행계획 출력값 해석'** 이었다). 축은 이렇게 판단한다:
- **빈도** — 무엇이 반복되나. 같은 실습·같은 수치·같은 명령이 여러 번 나오면 그게 축 후보.
- **중요도/무게** — 어디에 설명·시간·경고를 많이 썼나. 함정·강조(★·주의)가 몰린 곳.
- **귀결점** — 그날이 결국 무엇을 *하게* 만드나 (무엇을 읽고 / 판정하고 / 결정하나).
→ 이렇게 잡은 축이 **큰 그림 슬라이드의 주장 · 섹션 순서 · 각 슬라이드의 종속 관계**를 지배한다.
나머지 개념은 버리지 말고 **축을 뒷받침하는 사례로 종속**시켜 배치한다. **덱은 그 축이 발표의 중심이 되도록 설계.**

## 층 구조 (하루치 덱의 뼈대)
1. **큰 그림(holistic)** — 그날 개념이 어떻게 **한 뿌리(=축)에서 갈라지는지** 한 장. (예: 68일차 = pctfree 하나 → row migration·ITL·재구성)
2. **이전 수업과의 연결(context)** — 오늘이 **누적 커리큘럼의 어디에 얹히는지**. (예: 62~68 SELECT/DML 처리 흐름에서 오늘은 저장·블록 계층; ITL←64·66, full scan←63, pctfree←67) — 독립 주제 나열이 아니라 *쌓아온 흐름*으로 보이게.
3. **핵심 내용** — 개념마다 [주장 제목 + 다이어그램/표 + **핵심 메커니즘·근거·해결을 화면에**], 더 깊은 디테일만 노트.
4. **빈틈 & 궁금한 점** — 노트가 안 다룬 것·확인 필요·후속 질문. *이해했기에 던질 수 있는 질문.* 노트 밖 지식은 '확인 권장'으로 표시(지어내지 않음).

> **자습용 밀도 원칙**: 발표자가 말로 채우는 게 아니라 **혼자 읽고 이해하는 자료**이므로, 알맹이(메커니즘·근거·해결)를 노트에만 두지 말고 **화면에 구조화해 올린다**(Mayer 중복원리의 '내레이션 없음' 예외). 단 벽글 금지 — 다이어그램 + 3~5줄 요지. 다이어그램이 크면 `maxw`로 줄여 설명 자리를 확보.

## 적용한 원칙 → 구현

### A. Assertion-Evidence (Alley, Penn State)
- **제목 = 완전한 문장(주장)**, 주제어가 아니라 요지. sentence case.
  - ✗ "ITL 슬롯 부족" → ✓ "한 블록을 여러 세션이 고치는데 ITL 슬롯이 모자라면 마지막 세션이 대기한다"
- 본문 = **그 주장을 증명하는 시각자료**(다이어그램/표), 불릿 덤프 아님.
- 구현: `contentHeader`가 2줄 주장 헤드라인(19–24pt, 자동 축소) 렌더.

### B. Mayer 멀티미디어 학습원리 (교육 자료라 핵심)
- **중복(redundancy)·양식(modality)**: 화면에 읽을 문장을 두지 말고 **발표자 노트**로. → 슬라이드는 시각자료+키워드, 문장은 노트.
  - 구현: 스펙 `notes` → `slide.addNotes()`. (68일차 8개 노트)
- **신호(signaling)**: 색·화살표·번호로 요점 강조. Oracle Red 하나만 강조색. (ITL 도해: 초록=보유/빨강 점선=대기)
- **공간 근접(spatial contiguity)**: 라벨을 도해 **부품 옆에** 붙임(범례 분리 금지).
- **분절(segmenting)**: 복잡한 메커니즘은 steps/여러 슬라이드로.
- **멀티미디어**: 텍스트만 있는 슬라이드 금지 — 개념마다 시각자료.

### C. 시각설계 (Duarte·Reynolds·Tufte·CRAP)
- **신호 대 잡음**: 장식·그림자 남발·중복 텍스트 제거.
- **의도적 여백**: 내용이 짧으면 위에 붙이고 아래를 비우지 말고 **콘텐츠 밴드 수직 중앙**으로. → '사고성 여백'을 '프레임형 여백'으로.
  - 구현: `planContent`가 페이지별 slack을 계산해 중앙 정렬.
- **밀도 1-메시지/슬라이드**: 블록 몰아넣기 대신 한 슬라이드 한 요지. (저작 규약)
- **일관성**: 고정 그리드(좌여백 0.92·콘텐츠 1.9~6.6)·팔레트(neutrals + Oracle Red)·타입 스케일.
- **차트정크 제거**: 표는 헤더 룰만, 3D·그림자·격자 없음.

## 새 블록·필드 (슬라이드 스펙)
- `{"kind":"svg","svg":"<svg …>","caption":"…","h":opt}` — **자체 완결형 SVG**(인라인 fill/stroke, 외부 class 금지). `build_ppt`가 sharp로 PNG 래스터해 삽입(한글 렌더 OK). 비율 유지·중앙 배치. 높이는 viewBox 비율로 산정.
- content 슬라이드 `"notes":"…"` — 발표자 노트(상세 설명). 화면엔 안 나옴.
- content 슬라이드 `"title"` — **주장 문장**으로 작성(34자↑이면 자동 축소·2줄).

## 저작 체크리스트 (일차 덱)
1. 먼저 **주장 문장 목록**(각 슬라이드 헤드라인)을 쓴다 = 그게 스토리라인.
2. 슬라이드마다 **하나의 요지 + 하나의 시각자료(도해/표/코드)**. 문장 설명은 `notes`로.
3. 도해는 자체완결 SVG(팔레트 색 인라인), 라벨은 부품 옆에.
4. 큰 그림 1장 + 빈틈/질문 1장을 **반드시** 포함.
5. `preview.js`로 시각 QA(오버플로 0), 노트가 pptx에 들어갔는지 확인.

## 문체 — 'AI 티' 빼기 (은은하게)
원문 사실은 그대로 두고 **문장 버릇만** 손본다. 화면 텍스트(제목·콜아웃) 우선, 노트는 부차.
- **격언조 마무리 삭제** — "결국 핵심은 ~이다 / ~게 먼저다 / 정답은 취향이 아니라 ~". 지우거나 구체적 결과로 바꾼다.
- **헤지 상투구 → 조건** — "늘 최선은 아니다" → "20% 이상 읽으면 오히려 느리다"(실측 숫자로).
- **메타 나레이션 삭제** — "이 슬라이드는 ~를 보여준다 / 이해했기에 나오는 질문들".
- **강박적 대칭 깨기** — "A는 ~지만 B는 ~다"를 매 줄 반복하지 않기. 문장 길이를 섞는다(짧은 한 방 + 긴 설명).
- **한 줄 한 동사** — "명사 → 명사 → 명사" 금지. 화살표는 진짜 인과에만, 한 줄 1개.
- **교과서 정의체는 TERMS로** — 본문은 "X는 ~해서 ~한다"(기능 먼저).
- 톤: 딱딱한 문어체와 수다 사이 **중간 레지스터**. 완결형·정확한 용어·짧은 단어.

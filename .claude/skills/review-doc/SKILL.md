# /review-doc — 문서 정확성(사실) 검증

원본 학습 문서는 수업 중 직접 작성돼 **주제는 맞아도 세부 사실이 틀릴 수 있다**.
이 스킬은 문서를 **실제로 정독**해 사실 오류를 찾고, 원문 사실 범위에서 최소 교정한 뒤
`review_status.json` 의 정확성(accuracy)을 갱신한다. 기계 린터가 못 잡는 사각지대를 사람/LLM 판단으로 메운다.

인자: 대상 문서 (예: `/review-doc assets/rman_recovery.html`). 없으면 어느 문서인지 묻는다.

## 절차

### 1) 정독 (병렬 서브에이전트 권장)
개념(`h3.blk`, 없으면 `h2`)을 실제로 읽어 **세 범주**만 찾는다:
- **오류/혼동** — 사실과 다르거나, 내부 모순, 오해를 부르는 서술. (가장 중요)
- **얼버무림** — "여러 가지·대체로·적절히"로 구체(수치·이름·메커니즘)를 회피.
- **벽글** — 어려운 개념인데 도해·단계·비유 없이 산문뿐.
문서가 크면 개념 묶음별로 서브에이전트를 병렬로 돌려 결과를 종합한다.
쉬운 개념이 짧은 건 정상 — 지적하지 않는다.

### 2) 검증 (지어내지 않기)
찾은 '오류' 후보는 **정확한 Oracle 사실**로 교차 확인한다. 확신이 서는 것만 교정 대상.
불확실하면 **'확인 필요'로 표시**하고 임의로 바꾸지 않는다.

### 3) 최소 교정
- 원문 사실 범위에서 **틀린 문장만** 고친다(재작성 금지). 무엇을 왜 고쳤는지 한 줄.
- SVG 라벨도 틀렸으면 함께 고친다(예: 잘못된 소유 주체 라벨).
- 검은박스(미정의 SVG 클래스)가 있으면 대상 문서 `<style>`에 정의를 추가.

### 4) 배선·기록
```bash
python sync_and_verify.py --dir assets                       # 재주입·검증
python review.py accuracy <doc>.html --issues <수정건수> --method "LLM 정독"
python build_dashboard.py                                    # 대시보드 갱신
```
교정이 있었으면 사용자에게 **요약 보고**(개념·무엇이 틀렸고 어떻게 고쳤는지) 후,
필요 시 PPT 재생성. 끝에 **커밋/푸시**.

## 보고 형식
발견을 심각도순으로: `[문서·개념] 범주 — 틀린 서술 인용 → 교정`. 교정 못 한(확인 필요) 것은 따로 표시.
정독으로도 못 가리는 건 "원문 노트 대조가 필요"하다고 명시한다.

## 하지 말 것
- 근거 없이 사실을 '고치기'. 확신 없으면 확인 필요로 남긴다.
- 스타일·취향 지적(쉬운 개념이 짧다는 등)으로 잡음 만들기.

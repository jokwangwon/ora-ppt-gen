📚 핵심 노트 — 허브 안에 내장

수업 전체의 핵심을 이 페이지 안에 넣었어요. 외부 파일 없이 어디서든 펼쳐볼 수 있어요.

① 한 문장이 처리되는 흐름 (Part 0–5)
  • Parse — SQL 해시로 라이브러리 캐시 조회. 있으면 soft(재사용), 없으면 hard(옵티마이저가 실행계획 생성→라이브러리 캐시에 적재). 바인드 변수로 텍스트 고정해 soft 유도.
  • Bind — 바인드 변수에 실제 값 대입.
  • Execute(읽기) — DBA를 해시→bucket→체인(cache buffers chains latch)→버퍼 헤더→버퍼. 없으면 물리 I/O.
  • Execute(쓰기) — undo 확보→ITL 등록→row lock(TX X)→change vector→redo·undo 기록→블록 dirty.
  • Redo/Commit — WAL(redo 먼저). commit 시 LGWR write 대기 = log file sync.
  • Fetch — 결과(active set)를 사용자에게 반환.
② Lock & 동시성
  • DML = TM RX(3) + TX X(6) — 테이블엔 약한 표시, 그 행엔 강한 잠금(동시성 위해).
  • 모드: 0 null · 2 RS · 3 RX(DML) · 4 S(일반 인덱스 생성) · 5 SRX · 6 X(DDL).
상황결과
SELECT 중 dropTM null(0)→통과, SELECT는 ORA-08103
UPDATE 중 dropTM RX(3)→ORA-00054(busy)
일반 index 생성 중 DMLS(4)↔RX(3) 충돌 → enq: TM
deadlockORA-00060, 한 문장만 취소(트랜잭션은 직접 종료)
③ 저장구조 & 공간
  • 계층: block ⊂ extent ⊂ segment ⊂ tablespace(↔ datafile).
  • DMT(딕셔너리, enq: ST) vs LMT(비트맵, 표준).
  • FLM(freelist 수동→몰리면 buffer busy waits) vs ASSM(비트맵 자동, 기본).
  • pctfree 10% = ① row migration 방지 ② ITL 슬롯 공간. pctused 40% = insert 재사용(FLM).
  • initrans/maxtrans = ITL 슬롯 수/한도. 부족+pctfree 0 → enq: TX - allocate ITL entry(REQUEST=4).
  • HWM = full scan 범위. delete로 안 내려감, truncate가 리셋. 밀어올릴 때 enq: HW.
  • row migration / ITL 경합 해결 = alter table move(인덱스 UNUSABLE) + rebuild online.
④ Redo · DBWR · Checkpoint · 복구
  • undo=rollback+읽기일관성(CR) / redo=복구(roll forward).
  • WAL — dirty 쓰기 전 redo를 먼저. 그래서 LGWR가 DBWR보다 앞.
  • LGWR 트리거: commit·1/3·1MB·3초·DBWR 직전.
  • DBWR 트리거: free buffer 부족·dirty 임계·checkpoint·3초·tablespace offline.
  • Checkpoint — dirty 디스크 반영+위치 기록(CKPT). 복구 시간↓(자주↔평소 I/O 트레이드오프). 못 따라오면 log file switch (checkpoint incomplete).
  • 인스턴스 복구: Roll Forward(redo, checkpoint~장애) + Roll Back(undo, 미완료 트랜잭션).

⑥ SQL 튜닝 — 실행계획 읽기
  • 뽑기: set autotrace traceonly explain(예상) / gather_plan_statistics+display_cursor(실제, E-Rows vs A-Rows).
  • 읽기: 들여쓰기 가장 안쪽·위부터, 결과를 부모로.
  • 컬럼: Rows=예상 카디널리티, Cost=비교용, *=predicate. access(인덱스로 좁힘) vs filter(읽고 버림).
  • 액세스 경로: full scan(많이 읽을 때 유리) vs rowid scan(소량). "인덱스 항상 빠름"은 오해.
  • 체크 순서: ① 큰 테이블 full scan인데 소량? ② 핵심 조건이 filter만? ③ E≪A(추정 빗나감)? ④ 소량인데 HASH/대량인데 NL?
⑦ 백지 인출 체크리스트
  • SQL 한 문장 처리 과정을 프로세스·메모리부터 fetch까지 화살표로
  • 버퍼 캐시에서 블록 찾는 순서 (해시→?→?→?)
  • DML이 TM·TX를 동시에 잡는 이유
  • WAL을 한 문장으로 (왜 LGWR가 DBWR보다 먼저)
  • 인스턴스 복구 2단계와 각각 redo·undo 중 무엇
  • FLM vs ASSM, freelists 1이 buffer busy를 부르는 이유
  • row migration 해결책 (move+rebuild)
  • 실행계획 읽는 순서와 access vs filter 차이

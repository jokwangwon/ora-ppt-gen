복습
먼저 — Part 3에서 가진 것
이 위에 DBWR·Checkpoint를 얹는다

버퍼 캐시는 데이터파일 블록의 복사본을 담아 물리 I/O를 줄이는 SGA 영역. 핵심 흐름을 한 줄로:

  • 블록 찾기 — DBA를 해시 → bucket → 체인(cache buffers chains latch) → 버퍼 헤더 → 버퍼.
  • 관리 — working set의 LRU(재사용: hot/cold + 보조)와 LRUW(쓰기 대기: dirty buffer). touch count로 hot 승격.
  • 상태 — free / pinned / dirty(변경됨, 아직 디스크에 안 씀) / clean.

여기서 자연스러운 질문 — "dirty buffer는 결국 누가, 언제 디스크에 쓰지?" 그 답이 DBWR와 Checkpoint다.

PART A
DBWR — Database Writer
dirty buffer를 데이터파일에 기록하는 프로세스

DBWn(DBW0…)은 LRUW의 dirty buffer를 데이터파일에 내려쓰는 백그라운드 프로세스다. 핵심 성격은 "게으르다(lazy)" — I/O를 줄이려고 가능한 한 늦게, 여러 블록을 모아서 쓴다. 단, 아래 트리거가 오면 즉시 일한다.

A.1왜 DBWR보다 LGWR가 먼저인가 — WAL

DBWR가 dirty block을 데이터파일에 쓰기 전에, 그 변경의 redo는 이미 redo log file에 있어야 한다(Write Ahead Log). 그래서 DBWR는 그 dirty block의 redo가 아직 로그에 안 내려갔으면, LGWR가 그 redo를 먼저 써서 디스크에 안전해질 때까지 기다린 뒤에 쓴다(Part 4.5의 LGWR 5트리거 중 "DBWR 작동 직전"). WAL이 보장하는 건 "그 블록의 redo가 데이터보다 먼저 디스크에 있다"는 순서이지, 매번 LGWR를 새로 부른다는 뜻은 아니다.

왜 이 순서가 안전한가 만약 데이터파일(②)이 먼저 써지고 그 직후 장애가 나 redo(①)가 유실되면, 변경은 디스크에 있는데 그걸 되돌리거나 재현할 기록이 없어진다. redo를 항상 먼저 확보하면 "데이터파일은 늦게 써져도 redo로 복구 가능"이 보장된다. 그래서 LGWR는 부지런하고(commit마다) DBWR는 게을러도 된다.
핵심 대비 — LGWR: redo를 log file에. commit마다(부지런). DBWR: dirty block을 datafile에. 가능한 늦게·모아서(게으름). 둘의 순서를 묶는 규칙이 WAL.
PART B
Checkpoint — 체크포인트
"여기까지는 디스크에 안전하다"는 도장

Checkpoint는 특정 시점까지의 dirty buffer를 데이터파일에 모두 반영하고, 그 위치(checkpoint position)를 control file·datafile 헤더에 기록하는 사건이다. 실제 블록 쓰기는 DBWR가 하고, CKPT 프로세스는 그 위치 정보를 기록한다.

B.1왜 필요한가 — 복구 시간 때문

장애 후 복구는 "checkpoint position(가장 오래된 dirty buffer 시점) 이후의 redo만" 재생하면 된다. 그 앞은 이미 디스크에 안전하니까. incremental checkpoint에선 이 위치가 계속 앞으로 밀린다(딱 떨어지는 "마지막 checkpoint 순간"이 아니라 연속적으로 전진하는 지점). 그래서 checkpoint가 자주 = 복구가 빠름. 단 그만큼 DBWR가 자주 써서 평소 I/O 부담이 커진다 — 전형적인 트레이드오프.

B.3관련 대기 — log file switch (checkpoint incomplete)

redo 로그는 그룹을 돌려쓴다. 다음 그룹으로 넘어가려는데, 그 그룹의 redo가 아직 데이터파일에 반영 안 된 dirty buffer를 복구하는 데 필요하면(=checkpoint 미완료) 덮어쓸 수 없어 대기한다 — log file switch (checkpoint incomplete). 67일차 freelist 실습 때 이 대기가 찍혔던 이유가 이거다(대량 insert로 로그가 빨리 차서 checkpoint가 못 따라옴).

완화 — redo 로그 그룹을 더 크게/더 많이, 또는 DBWR가 더 부지런히 쓰게(checkpoint 강도↑). 단 checkpoint를 너무 세게 하면 평소 I/O가 늘어 균형이 필요하다.
PART C
인스턴스 복구
checkpoint·redo·undo가 만나는 곳

전원이 끊기는 등 비정상 종료가 나면, 메모리(버퍼 캐시·redo log buffer)의 내용이 날아간다. 재기동 시 오라클이 자동으로 두 단계로 일관성을 복원한다.

Roll Forward (Cache Recovery)
마지막 checkpoint부터 장애 시점까지의 redo를 재생해, commit 됐든 안 됐든 모든 변경을 메모리에 다시 만든다. redo가 있어 가능 — 그래서 WAL이 중요했다.
Roll Back (Transaction Recovery)
그중 commit 안 된 트랜잭션을 undo로 되돌린다. 그래서 undo의 목적(rollback)이 복구에서도 쓰인다.
한 그림으로 묶기 redo(앞으로 재생) + undo(뒤로 되돌림) + checkpoint(어디부터 재생할지의 시작점)가 인스턴스 복구에서 만난다. checkpoint가 복구의 출발선을 당겨주고, redo가 따라잡고, undo가 미완성 트랜잭션을 청소한다.
PART D
버퍼 캐시 더 보기
숙제에서 한 발 더 — 알아두면 좋은 것

D.1다중 버퍼 풀 — DEFAULT / KEEP / RECYCLE

한 캐시 안에서 모든 객체가 LRU를 공유하면, 한 번 쓰고 버릴 큰 테이블이 자주 쓰는 작은 테이블을 밀어낼 수 있다. 그래서 풀을 나눈다.

풀용도
KEEP항상 캐시에 두고 싶은 작고 자주 쓰는 객체 (db_keep_cache_size)
RECYCLE한 번 쓰고 버릴 크고 드물게 쓰는 객체 (빨리 내보냄)
DEFAULT나머지 일반 객체

객체에 storage(buffer_pool keep)로 지정. 여러 블록 크기를 쓰면 db_16k_cache_size처럼 블록 크기별 캐시도 따로 둔다.

D.2CR(Consistent Read) 블록 — 읽기 일관성

내가 SELECT하는 도중 남이 그 블록을 바꿔도, 나는 쿼리 시작 시점의 옛 모습을 봐야 한다. 오라클은 현재 블록(current)에 undo를 적용해 옛 버전(CR copy)을 메모리에서 재구성한다. 그래서 한 블록이 캐시에 current 1개 + CR copy 여러 개로 동시에 존재할 수 있다.

  • consistent gets — 읽기 일관성을 지킨 논리적 읽기 (대부분의 SELECT)
  • db block gets — current 모드 읽기 (주로 DML이 현재 값을 읽을 때)

D.3버퍼 캐시 히트율(BCHR)의 함정

히트율 = 논리 I/O 중 물리 I/O 없이 처리된 비율. 높다고 무조건 좋은 게 아니다 — 비효율 쿼리가 같은 블록을 캐시에서 수백만 번 읽으면(논리 I/O 폭주) 히트율은 99%여도 시스템은 느리다. 히트율보다 "논리 I/O 자체를 줄였는가"(불필요한 full scan·반복 읽기 제거)가 진짜 지표다.

면접 한 줄 — "버퍼 캐시 히트율이 높으면 좋은가?"에 "꼭 그렇지 않다. 논리 I/O가 과도하면 히트율이 높아도 비효율이다"라고 답하면 깊이를 보여준다.

Oracle DBA · 66일차 · 운영편

RMAN 백업·복구 치트시트

datafile을 잃었을 때 되살리는 절차 한 장. 진단 → offline → restore → recover → online, 그리고 nologging이 만든 함정과 dbms_repair 대처까지. 완전정복 가이드(내부구조편)와 짝을 이루는 운영편입니다.

STEP 0 전제 — Archive Mode 확인

미디어 복구(restore + recover)는 아카이브 로그가 있어야 가능하다. 먼저 모드를 확인한다.

SYS@ora19c> archive log list
Database log mode              Archive Mode
Automatic archival             Enabled
Current log sequence           65

Archive Mode + Automatic archival Enabled여야 datafile 복구가 가능하다.

STEP 1 백업 만들기

-- 스키마/데이터파일 구성 확인
RMAN> report schema;
-- 전체 백업(datafile을 백업셋으로 묶음)
RMAN> backup database;
-- 현재 redo 로그를 강제로 아카이브(백업 직후까지 매듭)
RMAN> alter system archive log current;
-- 백업/카피 목록, 오래된 백업 삭제
RMAN> list backup;
RMAN> list copy;
RMAN> delete backup;
무엇을 만드나 — 백업셋 vs 이미지 카피 backup database는 여러 datafile을 묶어 압축한 백업셋을 만든다(list backup으로 조회). backup as copy는 datafile을 그대로 복제한 이미지 카피다(list copy). 그리고 archive log current는 현재 redo 로그를 강제로 아카이브해, 백업 직후까지의 변경이 복구에 쓰일 수 있게 매듭짓는다 — 이게 빠지면 "백업은 했는데 그 직전 변경의 redo가 아직 안 잘려" 복구 범위에 구멍이 날 수 있다.

STEP 2 datafile 손실 → 복구 시나리오

datafile 7(users01.dbf)이 사라지면 startup이 ORA-01157: cannot identify data file 7로 실패한다. 시스템 외 datafile 하나만 문제일 때는 그 파일만 offline으로 내리고 나머지를 열어 서비스를 유지한 채 복구한다. 단 문제 파일이 SYSTEM·UNDO datafile이면 이 온라인 방식은 불가 — DB를 열 수 없으니 MOUNT 상태에서 전체를 restore·recover 한다.

진단 — Data Recovery Advisor
RMAN> list failure; → "One or more non-system datafiles are missing". list failure <ID> detail로 어느 파일인지 확인. (필요 시 advise failure → repair failure로 자동 복구 가능)
해당 datafile만 offline (MOUNT 상태에서)
먼저 startup mount로 MOUNT까지만 올린 뒤: RMAN> alter database datafile 7 offline; — open 실패(ORA-01157) 상태에선 아직 열 수 없으므로 MOUNT에서 문제 파일을 격리한다.
DB open (나머지 정상 서비스)
RMAN> alter database open; — 7번을 offline으로 격리했으니 이제 열린다. 7번을 제외한 테이블스페이스는 정상 사용
restore — 백업에서 파일 복원
RMAN> restore datafile 7; — 백업 조각에서 datafile을 디스크에 복원
recover — redo로 최신화
RMAN> recover datafile 7; — 백업 이후 변경을 아카이브/온라인 redo로 적용(media recovery)
online — 다시 투입
RMAN> alter database datafile 7 online; — 복구 완료, 정상 사용
핵심 공식 — datafile 단위 복구는 언제나 offline → (open) → restore → recover → online. restore는 "백업본 가져오기", recover는 "redo로 따라잡기"로 기억.

STEP 3 함정 — nologging 블록은 복구되지 않는다

복구를 끝내고 조회했더니 이게 터질 수 있다:

SYS@ora19c> select count(*) from hr.redo_table;
ORA-01578: ORACLE data block corrupted (file # 7, block # 560)
ORA-26040: Data block was loaded using the NOLOGGING option
왜 이런가 (65일차와 연결) nologging + /*+ append */(direct path)로 넣은 블록은 redo가 없다. 그래서 datafile을 restore·recover 해도 그 블록은 재구성할 redo가 없어 깨진 채로 남는다. "redo를 아끼면 그 데이터의 복구를 포기하는 것"이라는 트레이드오프를 직접 보여주는 장면. 복구가 중요한 테이블은 nologging을 쓰지 말거나, nologging 적재 후엔 즉시 백업을 다시 떠야 한다.

STEP 4 손상 블록 처리 — dbms_repair

깨진 블록을 건너뛰고 정상 데이터만 살린다.

순서명령효과
1exec dbms_repair.skip_corrupt_blocks(schema_name=>'HR', object_name=>'REDO_TABLE', object_type=>dbms_repair.table_object, flags=>dbms_repair.skip_flag)손상 블록을 건너뛰도록 표시 → 조회 시 무시
2alter table hr.redo_table move;정상 블록만 새 세그먼트로 옮겨 손상 블록을 실제로 떼어냄. skip은 조회 시 무시일 뿐이라, move로 옮겨야 손상 블록이 물리적으로 빠진다
3exec dbms_repair.skip_corrupt_blocks(..., flags=>dbms_repair.noskip_flag)skip 해제 → 이후 정상 조회

결과: 손상 블록을 제외한 정상 행만 남아 조회 가능(예: 107건).

복구가 아니라 포기다 — dbms_repair는 손상 블록의 행을 영구 포기하고 나머지만 살리는 응급 처치다. nologging으로 redo가 없어 잃은 데이터는 이 절차로도 되살아나지 않는다. 되살리려면 애초에 데이터 소스에서 다시 적재해야 한다.

참고 자주 쓰는 RMAN 명령

-- 구성/목록
RMAN> report schema;          -- datafile 구성
RMAN> list backup;            -- 백업셋 목록
RMAN> list copy;              -- 이미지 카피 목록
-- 진단 (Data Recovery Advisor)
RMAN> list failure;           -- 장애 목록
RMAN> list failure 85 detail;  -- 상세
RMAN> advise failure;          -- 복구 방안 제안
RMAN> repair failure;          -- 자동 복구 실행
-- datafile 복구 5종 세트
RMAN> alter database datafile 7 offline;
RMAN> alter database open;
RMAN> restore datafile 7;
RMAN> recover datafile 7;
RMAN> alter database datafile 7 online;
한 줄 요약 — 진단(list failure) → 격리(offline) → 서비스 유지(open) → 복원(restore) → 따라잡기(recover) → 투입(online). 그리고 nologging은 이 모든 복구를 무력화할 수 있다는 것.

Oracle RAC

지금까지는 서버 한 대 안에서의 이야기였다. RAC는 여러 서버(노드)가 하나의 데이터베이스를 공유하는 구조 — 한 대가 죽어도 서비스가 계속되고(고가용성), 노드를 더하면 처리량이 늘어난다(확장성). 이 문서는 그 구조와, 실습으로 클러스터를 세우는 과정을 쌓는다.

01
RAC란 — 여러 노드가 하나의 DB를 공유한다
shared-everything · 고가용성(HA) + 확장성(scalability) · 인스턴스 여럿, 데이터베이스 하나

RAC(Real Application Clusters) = 여러 서버(노드)에 각각 오라클 인스턴스를 띄우고, 그 인스턴스들이 하나의 데이터베이스(공유 스토리지의 데이터파일)를 함께 여는 구조. "인스턴스는 여럿, 데이터베이스는 하나"가 핵심 — 내부구조편에서 배운 인스턴스(메모리+프로세스)와 데이터베이스(디스크의 파일)의 분리가 여기서 결실을 맺는다.

얻는 것어떻게
고가용성(HA)노드 하나가 죽어도 남은 노드의 인스턴스가 같은 DB를 계속 서비스 — 무중단
확장성(scalability)노드를 추가하면 처리 능력이 늘어난다(scale-out) — 한 대를 키우는 scale-up과 대비
부하 분산SCAN 리스너가 접속을 여러 인스턴스로 분배
Cache Fusion — RAC의 심장두 인스턴스가 같은 블록을 캐시에 올릴 수 있으니, 한쪽이 바꾼 블록을 다른 쪽이 즉시 봐야 한다. 디스크를 거치지 않고 private 인터커넥트로 메모리→메모리 직접 전송하는 게 Cache Fusion이다 — 그래서 private 네트워크의 속도가 RAC 성능을 좌우한다(§2). 상세 동작은 뒤 차시에서.
02
네트워크 4종 — public · private · VIP · SCAN
각 IP가 맡은 역할이 다르다 · ping으로 확인하는 연결 상태

RAC는 노드마다 여러 IP를 쓴다 — 각자 역할이 다르다. 실습 구성(조광원 환경):

종류예 (rac1 / rac2)역할
Public192.168.56.190 / .191일반 클라이언트·관리 접속용 실제 IP
Private192.168.55.190 / .191인터커넥트 — 노드 간 Cache Fusion·클러스터 통신 전용(별도 서브넷 55)
VIP(가상 IP)192.168.56.192 / .193노드 장애 시 다른 노드로 떠서 클라이언트가 빠르게 재접속(TCP 타임아웃 회피)
SCAN192.168.56.194 (rac-scan)단일 접속점 — 클라이언트는 SCAN 하나만 알면 되고, 접속을 인스턴스들로 분배
$ ping -c 3 rac2        → 0% packet loss  (public 연결 OK)
$ ping -c 3 rac2-priv   → 0% packet loss  (private 인터커넥트 OK)
$ ping -c 3 rac2-vip    → 100% packet loss, Destination Host Unreachable
   # VIP는 클러스터웨어(Grid)가 떠야 활성화된다 — 설치 전이라 아직 unreachable(정상)
왜 VIP가 필요한가노드가 죽었는데 물리 IP로 접속을 시도하면, 클라이언트는 TCP 타임아웃(수십 초)을 기다린 뒤에야 실패를 안다. VIP는 노드 장애 시 살아있는 노드로 즉시 떠서(failover) "여긴 지금 없다"를 바로 알려주니, 클라이언트가 대기 없이 다른 노드로 재접속한다. 설치 전 ping이 안 되는 건 아직 클러스터웨어가 VIP를 안 띄웠기 때문 — 오류가 아니다.
SCAN(Single Client Access Name). 노드가 늘거나 줄어도 클라이언트 접속 문자열을 안 바꾸게 해주는 단일 이름. 클라이언트는 rac-scan 하나만 알면 되고, SCAN 리스너가 그 뒤의 인스턴스들로 연결을 분배한다 — 노드 추가/제거가 클라이언트에 투명해진다.
03
소프트웨어 스택 — Grid Infrastructure와 DB
Clusterware + ASM = Grid · 그 위에 Oracle Database · 설치 순서에는 이유가 있다

RAC는 두 개의 소프트웨어(홈)를 순서대로 깐다:

  • Grid Infrastructure(GRID_HOME /u01/app/19.0.0/grid) — Clusterware(노드를 클러스터로 묶고 VIP·SCAN·리스너·DB 같은 리소스를 관리) + ASM(공유 스토리지 관리). ASM 인스턴스 이름은 노드별 +ASM1·+ASM2.
  • Oracle Database(ORACLE_HOME /u01/app/oracle/product/19.0.0/dbhome_1) — 실제 DB 소프트웨어. 인스턴스 racdb1·racdb2가 데이터베이스 racdb를 연다.
왜 Grid를 먼저 까나DB가 공유 스토리지에 데이터파일을 두려면 그 스토리지(ASM)가 먼저 있어야 하고, 여러 노드가 하나의 DB를 열려면 그들을 묶는 클러스터(Clusterware)가 먼저 있어야 한다 — 그래서 Grid → DB 순서다. 환경변수도 둘을 분리한다: .grid_env(ORACLE_SID=+ASM1)와 .db_env(ORACLE_SID=racdb1)를 alias(grid_env/db_env)로 전환.
04
공유 스토리지와 ASM — 디스크를 함께 쓴다
shareable 디스크 · oracleasm 드라이버 · 디스크 그룹(DATA/FRA) · redundancy

RAC의 전제는 모든 노드가 같은 디스크를 볼 수 있어야 한다는 것. VirtualBox 실습에선 디스크를 shareable로 만들어 두 VM에 물린다:

# (윈도우 호스트) 5GB 고정 디스크 6개 생성 후 공유 가능으로
> VBoxManage createhd --filename C:\rac\asm1.vdi --size 5120 --format VDI --variant fixed
> VBoxManage modifyhd C:\rac\asm1.vdi --type shareable   # asm2~6도 동일

리눅스에선 oracleasm 드라이버로 이 디스크들을 ASM이 쓸 형태로 준비한다 — 디스크에 라벨(ASMDISK01…)을 찍는 과정:

# oracleasm configure -i     # owner=oracle, group=oinstall, on-boot=y
# oracleasm init             # /dev/oracleasm 마운트, 드라이버 로드
# fdisk /dev/sdb             # n→enter×4→w : 디스크 전체를 파티션 1개로 (sdb1)
# oracleasm createdisk ASMDISK01 /dev/sdb1   # sdc1~sdh1도 → ASMDISK07까지
# oracleasm scandisks        # 반대 노드(rac2)에선 이것만 — 라벨을 인식
# oracleasm listdisks        ASMDISK01 … ASMDISK07
라벨은 한 노드에서만 찍는다. createdisk는 rac1에서만(디스크에 라벨을 쓰는 작업). rac2는 scandisks만 하면 같은 공유 디스크의 라벨을 그대로 인식한다 — "Instantiating disk ASMDISK01…". 공유 디스크라서 rac2가 라벨을 다시 쓸 필요가 없다.

ASM은 이 디스크들을 묶어 디스크 그룹으로 관리한다 — 파일 시스템 대신 오라클 전용 볼륨 관리자:

디스크 그룹용도
DATA데이터파일·컨트롤파일·redo 등 실데이터 (Grid 설치 때 생성)
FRA(Fast Recovery Area)백업·아카이브 로그·플래시백 (ASMCA로 추가 생성)
redundancy — 미러링의 정도디스크 그룹 생성 시 이중화 수준을 고른다: external(미러링 없음 — 스토리지 장비가 이미 RAID, 실습은 이것) · normal(2중 미러) · high(3중 미러). ASM이 소프트웨어로 블록을 여러 디스크에 복제해 디스크 하나가 죽어도 견디게 한다. 실습은 external(디스크 1~2번만 골라 DATA 구성).

06
클러스터 검증 — crsctl로 리소스를 읽는다
crsctl stat res -t · 두 인스턴스가 같은 racdb를 Open · oratab

설치가 끝나면 crsctl stat res -t로 클러스터가 관리하는 리소스들의 상태를 본다 — 무엇이 어느 노드에서 ONLINE인지:

$ crsctl stat res -t
Local Resources           (노드마다 하나씩)
  ora.LISTENER.lsnr        rac1 ONLINE · rac2 ONLINE   ← 로컬 리스너
  ora.net1.network / ora.ons / ora.chad
Cluster Resources         (클러스터 전체에서 관리)
  ora.DATA.dg              rac1·rac2 ONLINE            ← DATA 디스크 그룹
  ora.FRA.dg               rac1·rac2 ONLINE            ← FRA 디스크 그룹
  ora.asm                  rac1·rac2 Started,STABLE    ← ASM 인스턴스
  ora.LISTENER_SCAN1.lsnr  rac1 ONLINE                 ← SCAN 리스너(클러스터에 1개)
  ora.rac1.vip / ora.rac2.vip / ora.scan1.vip         ← VIP·SCAN VIP
  ora.racdb.db             rac1 Open · rac2 Open       ← DB: 두 인스턴스가 Open!

마지막 ora.racdb.db가 rac1·rac2 양쪽에서 Open — 이것이 RAC가 섰다는 증거다. 각 노드에서 접속해 인스턴스 이름을 확인하면 서로 다르다:

rac1$ sqlplus / as sysdba   →  SYS@racdb1> select instance_name from v$instance;  racdb1
rac2$ sqlplus / as sysdba   →  SYS@racdb2> select instance_name from v$instance;  racdb2
   # 인스턴스는 racdb1·racdb2로 다르지만, 여는 데이터베이스(racdb)는 하나
oratab과 oraenv — SID 전환/etc/oratab에 +ASM1·racdb1(rac2는 +ASM2·racdb2)을 등록해 두면 . oraenv로 ORACLE_SID·ORACLE_HOME을 한 번에 전환한다. 한 노드에서 ASM(+ASM1)과 DB(racdb1) 두 인스턴스를 오가야 하므로 이 전환이 일상 — crsctl은 클러스터 전체 상태를, v$instance는 지금 붙은 인스턴스를 본다.
OFFLINE 3번은 정상. 리소스 목록에서 ora.asmgroup의 3번이 OFFLINE으로 보이는 건 2노드 구성이라 세 번째 슬롯이 비어 있는 것 — 장애가 아니다. rac1·rac2(1·2번)가 ONLINE이면 정상.

01
SQL Tuning이란
시작은 SQL의 의미를 파악하는 것

SQL Tuning은 SQL문을 최적화하여 빠른 시간 안에 원하는 결과값을 얻기 위한 작업이다. 그 출발점은 도구나 힌트가 아니라 — SQL의 의미(작성 의도)를 제대로 파악하는 것이다. "이 쿼리가 진짜 원하는 게 무엇인가, 얼마나 많은 데이터를 건드려야 하는가"를 알아야 올바른 처리 방법을 고를 수 있다.

전제 — 튜닝은 내부구조편의 지식 위에서 동작한다. 블록·HWM·버퍼 캐시·인덱스 구조를 알아야 "왜 이 실행계획이 느린가"가 보인다. 여기서는 그 지식을 활용하는 쪽에 집중한다.

02
실행계획 읽기 — 모든 튜닝의 기반
계획을 뽑고, 순서대로 읽고, 느린 지점을 찾는다

튜닝의 첫 동작은 인덱스를 만드는 게 아니라 실행계획을 읽는 것이다. "옵티마이저가 데이터를 어떤 경로로·어떤 순서로 가져오는가"를 못 읽으면 무엇을 고칠지 알 수 없다. 이 장이 안 되면 뒤의 모든 튜닝이 공허하다.

2.1실행계획 뽑는 3가지 방법

방법명령성격
autotraceset autotrace traceonly explain예상 계획만 (실행 안 함)
EXPLAIN PLANexplain plan for …; → select * from table(dbms_xplan.display);예상 계획 (실행 안 함)
display_cursor/*+ gather_plan_statistics */ 실행 후 → dbms_xplan.display_cursor(format=>'ALLSTATS LAST')실제 실행 계획 + 실측치
예상 vs 실제 — 가장 중요한 구분 explain plan·autotrace explain은 옵티마이저의 예상(E-Rows)만 보여준다. 반면 display_cursor는 실제로 돌린 뒤 A-Rows(실제 행 수)까지 보여준다. E-Rows와 A-Rows가 크게 차이 나면 옵티마이저가 데이터를 잘못 추정한 것 — 잘못된 계획의 가장 흔한 원인이다. 그래서 진짜 튜닝은 display_cursor로 한다.

2.2읽는 순서 — 안쪽·위부터

실행계획은 트리다. 들여쓰기가 가장 깊은(안쪽) 라인부터, 같은 깊이면 위에서 아래로 실행되고, 자식이 만든 결과를 부모에게 올린다.

컬럼의미 · 볼 점

2.5예제로 읽어보기

2.2의 계획을 실제로 찍으면 이렇게 나온다. display_cursor로 E-Rows(예상)와 A-Rows(실제)를 같이 보는 게 핵심이다.

SYS@ora19c> select /*+ gather_plan_statistics */ e.last_name, d.department_name
          from hr.employees e join hr.departments d using(department_id)
          where e.department_id = 50;
SYS@ora19c> select * from table(dbms_xplan.display_cursor(format=>'ALLSTATS LAST'));

----------------------------------------------------------------------
|  0 | SELECT STATEMENT              |             |        |     45 |
|  1 |  NESTED LOOPS                 |             |     45 |     45 |
|* 2 |   TABLE ACCESS BY INDEX ROWID | EMPLOYEES   |     45 |     45 |
|* 3 |    INDEX RANGE SCAN           | EMP_DEPT_IX |     45 |     45 |
|  4 |   TABLE ACCESS BY INDEX ROWID | DEPARTMENTS |      1 |     45 |

   3 - access("E"."DEPARTMENT_ID"=50)

  • E-Rows = A-Rows (45 = 45) → 옵티마이저 추정이 정확하다. 괴리가 없으니 통계는 건강하다.
  • 핵심 조건이 access(... =50) → 인덱스로 바로 범위를 좁혔다(좋음). filter가 아니다.
  • 50번 부서 45명만 인덱스로 집어 각자 부서를 unique scan으로 붙였다 — 소량이라 NESTED LOOPS가 적합.
  • 안쪽(Id 4·5)의 E-Rows 1은 1회 기준, A-Rows 45는 바깥 45행만큼 반복된 누적치(§2.6 ④의 E×Starts 규칙).
무엇을 의심하나 (체크 순서) ① 큰 테이블에 TABLE ACCESS FULL인데 소량만 찾나? → 인덱스 기회. ② 핵심 조건이 filter로만 처리되나? → 인덱스 기회. ③ E-Rows ≪ A-Rows로 추정이 크게 빗나갔나? → 통계 갱신·카디널리티 문제. ④ 소량인데 HASH JOIN, 대량인데 NESTED LOOPS? → 조인 방식 의심.
홈랩에서 바로 — ora19c에서 set autotrace on 또는 위 gather_plan_statistics + display_cursor로 직접 찍어보세요. 자료를 읽는 것보다 내 쿼리의 계획을 눈으로 보는 게 훨씬 빨리 늡니다.

03
데이터 액세스 경로
무엇을 어떻게 읽을지 — 튜닝의 핵심 갈림길

옵티마이저가 데이터를 가져오는 방법은 크게 둘이다. 어느 쪽이 빠른지는 "얼마나 많은 행을 읽느냐"에 달렸다.

경로설명유리한 경우
full table scan세그먼트 첫 블록부터 HWM까지 통째로 읽음많은 데이터 검색
rowid scan · by user rowid사용자가 rowid를 직접 지정해 한 행을 콕 집음rowid를 이미 알 때
rowid scan · by index rowid인덱스를 타고 rowid를 얻어 해당 블록의 행을 읽음소량 검색
직관 · full scan vs index의 분기점 전체의 작은 일부만 찾을 땐 인덱스(by index rowid)가 빠르다. 하지만 전체의 상당 비율을 읽어야 한다면, 인덱스를 타며 블록을 하나씩 random하게 읽는 것보다 full scan으로 multi-block I/O로 쓸어 담는 게 오히려 빠르다. "인덱스가 항상 빠르다"는 흔한 오해다 — 읽어야 할 양이 기준이다.
04
Full Table Scan
첫 블록부터 HWM까지, multi-block로

full table scan은 세그먼트의 첫 블록부터 HWM(High Water Mark, 사용 블록과 미사용 블록의 경계)까지 읽는다. 핵심은 한 번의 I/O call에 db_file_multiblock_read_count(이 환경 =78)만큼의 블록을 묶어 읽는 Multi Block I/O라는 점. 그래서 db file scattered read 대기가 발생한다.

인덱스가 없으면 조건을 줘도 full scan이 된다. hr.employees를 복사한 hr.emp(인덱스 없음)에 PK 같은 조건을 줘도 옵티마이저가 다른 길이 없어 TABLE ACCESS FULL을 택한다.

SYS@ora19c> create table hr.emp tablespace users as select * from hr.employees;
SYS@ora19c> set autot traceonly exp
SYS@ora19c> select * from hr.emp where employee_id = 100;

--------------------------------------------------------------
| Id  | Operation         | Name | Rows | Bytes | Cost (%CPU)|
--------------------------------------------------------------
|   0 | SELECT STATEMENT  |      |    1 |    69 |     3   (0)|
|*  1 |  TABLE ACCESS FULL| EMP  |    1 |    69 |     3   (0)|
--------------------------------------------------------------
   1 - filter("EMPLOYEE_ID"=100)
튜닝 포인트 — 단 한 행(employee_id=100)을 찾는데 테이블 전체를 HWM까지 읽고 있다. 여기에 employee_id 인덱스를 만들면 by index rowid로 한 행만 집어 비용이 급감한다. "한 행을 위해 full scan"이 바로 튜닝 대상의 전형이다.
05
rowid scan & 인덱스
소량을 콕 집는 길 — 그리고 인덱스를 못 타는 함정

5.1rowid scan — 주소로 한 행

rowid는 행의 물리 주소다. rowid를 직접 주면 옵티마이저는 인덱스도 안 거치고 그 블록으로 직행한다 — TABLE ACCESS BY USER ROWID(Cost 1). 인덱스를 경유해 얻은 rowid로 읽으면 TABLE ACCESS BY INDEX ROWID다. 둘 다 single block I/O라 db file sequential read가 발생한다.

형식구성
extended (v8, 18자)데이터오브젝트(6) + 파일#(3) + 블록#(6) + 행#(3) · 예 AAAS2nAAHAAAAD0AAD
restricted (v7)블록#(8) . 행#(4) . 파일#(4) · 예 000000F4.0003.0007

dbms_rowid.rowid_object/relative_fno/block_number/row_number로 rowid를 분해해 "이 행이 몇 번 파일·블록·행에 있나"를 볼 수 있다.

5.2인덱스 스캔의 종류

인덱스는 root → branch → leaf 트리다. range scan은 leaf에서 시작 값을 찾고, 옆 leaf로 이어가며 "하나 더 있나?"를 확인하는 one-plus-one 방식으로 끝을 판단한다.

스캔언제
INDEX RANGE SCAN범위·비유일. 같은 값이 더 있나 leaf를 이어 확인
INDEX UNIQUE SCAN유일 인덱스 + =일 때만. 범위(<=)면 RANGE로 떨어진다
INLIST ITERATORin (a,b,c) — 값 수만큼 인덱스 탐색 반복
INDEX FULL SCAN인덱스만으로 충당될 때(예: 인덱스 컬럼만 SELECT)
AND-EQUAL (index merge)두 단일 인덱스의 rowid 결과를 교집합(and_equal 힌트)

5.3★ 인덱스를 "안 타는" 경우

인덱스가 있어도 옵티마이저가 버리는 패턴들. 면접·시험 단골이다.

조건계획이유

핵심 원칙 인덱스 컬럼을 가공(함수·연산·형변환)하면 그 순간 인덱스는 죽는다. 가공이 필요하면 컬럼이 아니라 상수 쪽을 바꾸거나, 함수 기반 인덱스를 따로 만든다. 그리고 filter로만 처리되는 핵심 조건이 보이면 인덱스(또는 결합 인덱스) 기회다.

5.4결합(조합) 인덱스 — 선두 컬럼이 전부

자주 함께 쓰는 컬럼은 하나의 결합 인덱스로 묶는다. (last_name, first_name)를 만들면 두 조건이 한 번의 access로 처리된다(앞의 filter 문제 해소). 단 선두 컬럼 선택이 성패를 가른다.

왜 선두 컬럼이 중요한가 — 전화번호부 결합 인덱스 (last_name, first_name)는 전화번호부처럼 성(last)으로 먼저 정렬하고, 같은 성 안에서만 이름(first)으로 정렬돼 있다. 그래서 성을 알면 한 구간으로 좁혀지지만, 성을 모르고 이름만 알면 그 이름이 책 전체에 흩어져 있어 정렬이 무용지물이다.

그래서 선두 컬럼은 이런 컬럼을 골라야 한다:

  • 혼자서도 자주 쓰이는 컬럼을 앞에 (선두가 조건에 없으면 위처럼 흩어져 못 좁힘)
  • 범위를 많이 줄여주는(선택도 높은) 컬럼을 앞에
  • = 비교로 쓰는 컬럼을 앞에 — 범위(BETWEEN·>) 조건을 선두에 두면, 그 구간 안에서 후행 컬럼은 연속 정렬이 아니라서 뒤 컬럼으로 더 못 좁힌다

비트맵이 뭐길래 — 값 하나마다 "그 값을 가진 행은 1, 아니면 0"인 비트벡터를 만든다. 값 종류가 적을수록(저선택도) 벡터가 몇 개 안 되고 압축이 잘 돼 작다.

AND/OR이 왜 싼가 — index_combine의 정체 두 조건을 겹칠 때 bitmap은 두 비트벡터를 bitwise AND/OR만 하면 된다(gender='m' AND dept=10 → 두 벡터에서 나란히 1인 자리만 남긴다). §5.6에서 index_combine이 and_equal(10)보다 싼 4였던 이유가 이것 — 옵티마이저가 B*tree의 rowid를 비트맵으로 변환(BITMAP CONVERSION FROM ROWIDS)해 비트 연산으로 교집합하기 때문이다. rowid를 하나씩 맞춰보는 AND-EQUAL보다 비트 겹치기가 싸다. 확인 권장

6.3버퍼 피닝 (Buffer Pinning)

한 줄 비유 — 책 45권을 꺼내는데, 여러 권이 같은 책장에 꽂혀 있다. 책장 하나를 꺼낼 때마다 제자리에 도로 꽂지 않고 옆에 둔 채(=핀) 계속 빼면, 책장까지 걸어가는 횟수가 확 준다. 여기서 "책장까지 걸어가기"가 블록 읽기(논리 I/O)다.

이 기법이 왜 필요한지를 문제 → 원인 → 해결 순으로 보자.

① 문제 — 인덱스 읽기는 사실 '작은 NL 루프'

Id 2 ↔ Id 1 의 반복 실행계획의 두 줄이 이렇게 주고받는다.
  1. Id 2 (INDEX RANGE SCAN): 인덱스 leaf에서 조건에 맞는 rowid 하나를 찾아 Id 1에 넘긴다.
  2. Id 1 (TABLE ACCESS BY INDEX ROWID): 그 rowid의 테이블 블록을 읽어 결과 한 행을 만든다.
  3. 다시 Id 2가 다음 rowid → Id 1이 또 읽고 … 행 수(45)만큼 반복.
순진하게 보면 행마다 테이블 블록을 한 번씩 읽으니 45번 + 인덱스 ≈ 46번이다.

② 알고보니 + ③ 그래서 핀 — 블록 단위로 보면

그런데 같은 부서원(department_id=50) 행들은 테이블에서 몇 개의 블록에 몰려 있다. Id 1이 한 번 읽은 블록을 바로 풀지 않고 핀으로 붙들어 두면, 다음 rowid가 같은 블록을 가리킬 때 다시 안 읽고 재사용한다.

핀이 정확히 뭔가 — 무엇을·왜·언제 푸나
  • 무엇 — 핀(buffer pin)은 그 블록의 관리 카드(buffer header)에 꽂는 "지금 사용 중" 표시다. 데이터 본문이 아니라 블록마다 붙은 머리표에 기록된다.
  • 왜 절약되나 — 핀이 살아 있으면 그 블록은 메모리에 그대로 있어, 다음에 같은 블록을 쓸 때 cache buffers chains latch를 잡고 해시 체인을 뒤지는 과정을 통째로 건너뛴다. 즉 "주소로 다시 찾아가는" 게 아니라 "이미 손에 쥔" 것이다. 논리 I/O(Buffers)와 latch 점유가 함께 준다.
  • 언제 풀리나 — 다른 블록으로 넘어갈 때, 또는 그 operation/fetch 단위가 끝날 때 푼다. 그래서 같은 블록이 연속으로 와야 효과가 있고, 중간에 다른 블록이 끼면 풀려서 절약이 사라진다(= clustering이 나쁘면 효과 없음).
효과를 키우는 두 가지
  • BATCHED — rowid를 모아 블록별로 묶어 접근(TABLE ACCESS BY INDEX ROWID BATCHED). 같은 블록을 한 번에 처리해 피닝 효과를 키운다.
  • clustering factor — 같은 키의 행들이 테이블에서 가까이 모여 있을수록(클러스터링 팩터가 좋을수록) 연속 rowid가 같은 블록에 떨어져 피닝 효과가 커진다. 흩어져 있으면 매 행이 다른 블록이라 절약이 사라진다.
39일차 Execute에서 본 buffer pin이 여기서 튜닝 효과(논리 I/O 절감)로 다시 등장한 셈이다.

6.7버퍼 캐시 히트율(BCHR) — 높다고 좋은 게 아니다 72일차

BCHR(Buffer Cache Hit Ratio)은 논리 읽기 중 디스크로 안 간 비율이다.

BCHR = ( 1 - disk / (query + current) ) * 100
   query + current = 논리 읽기(블록),   disk = 물리 읽기(블록)
히트율이 높은데 느리다면 — 지표를 잘못 본 것 "BCHR은 높을수록 좋다"고 배우지만, 나쁜 실행계획이 오히려 BCHR을 100%에 가깝게 만든다. 실행계획이 나빠 논리 읽기(query)가 폭증하면 분모가 커져 disk/(query+current)가 0에 수렴 — 디스크는 안 가고 캐시에서만 잔뜩 읽어서 히트율은 오히려 올라간다. 그래서 봐야 할 것은 히트율(비율)이 아니라 논리 I/O(Buffers·consistent gets)의 절대량이다. §6.2의 "튜닝은 보통 consistent gets를 본다"와 같은 결론이다.

숫자로 보면 함정이 분명하다 — 같은 결과를 내는 두 계획 (이해용 예시 수치):

BCHR99.95%90.9%
실제 한 일(논리 I/O)✗ 500배 많음✓ 적음
히트율이 높은 쪽이 오히려 500배 느리다 — 나쁜 계획은 캐시에서 10만 블록을 헛읽고 디스크(50)는 거의 안 가니 1-50/100000 ≈ 99.95%. 좋은 계획은 200블록만 읽고 끝나지만 디스크 비중이 커 1-20/220 ≈ 90.9%. 비율(BCHR)이 아니라 논리 I/O 절대량(query)을 봐야 진짜 빠른 쪽을 고른다.
BCHR은 어디서 나오나 · 대신 무엇을 보나 시스템 전체 BCHR은 v$sysstat의 consistent gets·db block gets·physical reads로 계산된다 — 인스턴스 누적이라 SQL 하나의 문제를 더 뭉뚱그린다. 그래서 진단은 BCHR을 지나쳐 아래로 내려가 본다: ① SQL별 Buffers·consistent gets(§6.2), ② A-Time·대기(10046, §6.6), ③ E-Rows↔A-Rows 괴리(§2.5). 확인 권장 — BCHR 무용론은 널리 알려진 표준 논지이니 세부는 직접 확인 권장.

7.1NL은 바깥 각 행마다 안쪽을 반복한다

Nested Loop은 바깥(driving) 테이블의 각 행마다 안쪽을 인덱스로 조회해 붙인다. 그래서 바깥이 작을수록 안쪽 반복이 적어 싸다. 반복 횟수는 실행계획의 Starts에 그대로 찍힌다.

driving과 driven — 왜 그렇게 부르나 이중 for문으로 보면 이름이 그대로 풀린다. for each row in OUTER { probe INNER } —
  • driving(=outer, 선행): 바깥 루프. 자기 행 수만큼 안쪽을 돌려 반복을 민다. 그 횟수가 곧 Starts다. 그래서 옵티마이저는 적게 걸러진 작은 쪽을 driving으로 두려 한다.
  • driven(=inner, 후행): 안쪽 루프. 스스로 횟수를 못 정하고 driving이 주는 만큼 끌려서 조회당한다(수동태). 매 outer 행의 조인키로 lookup된다.
그래서 driven엔 인덱스가 필수 — driven은 outer 행마다 반복해 뒤져지므로, 조인 컬럼에 인덱스가 없으면 Starts번 × full scan으로 터진다. NL 튜닝의 정석은 ① driving은 작은(많이 걸러진) 쪽, ② driven의 조인 컬럼엔 인덱스. 아래 두 플랜에서 driven이 늘 인덱스(UNIQUE/RANGE SCAN)로 접근되는 이유다.
헷갈리지 말 것 — outer 테이블 ≠ outer 조인 여기서 outer 테이블은 NL 바깥 루프(=driving)를 말한다. 짝 없는 행을 NULL로 살리는 outer join(외부 조인 · LEFT/RIGHT)과는 이름만 같고 다른 개념이다.

7.2실측 ① 바깥이 1행 — employee_id=100

Id 옆의 *와 Predicate Information — 무엇을 알려주나 *는 "이 단계에 조건(predicate)이 걸려 있으니 아래를 보라"는 포인터다. 별표 있는 Id(*3·*5)만 아래 Predicate Information에 조건이 나오고, 없는 Id(0·1·2·4)는 순수 작업(조인·테이블 접근)일 뿐이다. 아래에서 읽는 것은 어느 단계가 어떤 조건을, 그리고 access(인덱스로 좁힘) 인지 filter(읽고 버림) 인지다(§2.3).

  • Id 5 access(E.DEPARTMENT_ID = D.DEPARTMENT_ID) — 이건 조인 조건이다. driven(DEPARTMENTS)을 outer 행의 department_id로 인덱스 lookup했다는 증거 — 앞의 "driven엔 인덱스"가 여기서 access로 확인된다. 만약 여기가 filter였다면 조인키 인덱스가 없어 매 반복마다 훑었다는 뜻(터짐 신호).
실전에선 핵심 조건이 filter로만 잡히거나 filter(TO_NUMBER("X")=100)처럼 함수가 씌워져 나오면, 인덱스를 못 타는 원인이 여기서 그대로 드러난다.

7.9Sort Merge Join — 정렬 두 번, 병합 한 번 74일차

NL은 건건이 버퍼 캐시를 오가는 random access 위주라 대용량에서 한계가 있다. Sort Merge는 ① 양쪽 집합을 조인 키로 각각 정렬하고 ② 정렬된 둘을 merge한다. 건수가 많을 때 유리하고, 대신 sort 부하를 감수한다. 정렬은 PGA의 sort area에서 하므로 SGA를 경유하는 접근보다 빠르다. 힌트 use_merge.

| 5 |    TABLE ACCESS FULL EMP     | Starts 1 |          3 |

7.10Hash Join — 작은 쪽으로 해시 테이블, 큰 쪽으로 probe 74일차

NL은 대량에서 random access가 부담, SM은 정렬이 부담 — 그럴 때 Hash Join. 작은 집합을 build table(해시 테이블)로 만들어 hash_area_size 영역(수동 PGA 관리 파라미터 — 기본인 자동 관리에선 pga_aggregate_target 기반 작업영역이 대신함)에 올리고, 큰 집합을 probe하며 해시 탐색으로 조인한다. 해시 키 컬럼에 중복이 적을수록 유리하고(같은 값이 몰리면 그 버킷을 길게 뒤진다 — 다른 값이 같은 버킷에 드는 해시 충돌과는 별개 개념, 아래 60·210이 충돌의 예), = 조건에서만 쓸 수 있다. 힌트 use_hash.

-- leading(d,e) use_hash(e) — Buffers 13
|*1 |  HASH JOIN         |      | 13 | Used-Mem 1500K |
| 2 |   TABLE ACCESS FULL| DEPT |  2 |  ← build (작은 쪽 27행)
| 3 |   TABLE ACCESS FULL| EMP  | 10 |  ← probe (큰 쪽 107행)

SYS> select ora_hash(department_id,10), department_id, department_name from hr.dept order by 1;

  1  210  IT Support     (버킷 하나에 여럿 = 충돌도 있다)
SYS> select ora_hash(department_id,10), department_id, employee_id, last_name from hr.emp order by 1;
  1   60  103 Hunold  ← emp의 60번 사원들도 같은 버킷 1로 찾아간다
동작 그림 ① DEPT 27행을 ora_hash(department_id)류의 해시로 버킷에 분류해 메모리에 올린다(build). ② EMP 107행을 하나씩 읽으며 같은 해시 함수를 통과시켜 해당 버킷만 뒤진다(probe) — 정렬도, 반복 인덱스 탐색도 없다. 조인 자체는 메모리에서 끝나 Buffers는 두 full scan(2+10)+α다. 등호(=)가 아니면 해시 버킷 매칭이 성립하지 않아 못 쓴다.

7.11build를 바꿔치는 옵티마이저 — swap_join_inputs 74일차

3테이블 해시(leading(d,e,l))를 돌렸더니, 어떤 실행에선 LOC이 위(build)로 올라간 플랜이 나왔다. 순서 힌트를 줬는데도 옵티마이저가 해시의 build/probe만 맞바꾼 것 — 조인 순서는 유지하되 "누구를 메모리에 올릴지"는 더 작은 쪽으로 스스로 바꾼다.

-- 같은 leading(d,e,l)인데:  ①                          ②
|1| HASH JOIN               |1| HASH JOIN     ← LOC이 build로 승격
|2|  HASH JOIN (d build)    |2|  TABLE ACCESS FULL LOC
|5|  TABLE ACCESS FULL LOC  |3|  HASH JOIN (d⋈e 결과가 probe)
-- 원위치 강제: no_swap_join_inputs(l) / 반대 강제: swap_join_inputs
outer join은 순서가 문법으로 강제된다 — (+)가 붙지 않은 테이블이 항상 선행 — 문법이 강제하는 건 조인 순서이고, build/probe는 비용 판단이라 옵티마이저가 스스로 RIGHT OUTER로 바꿔 (+)쪽을 build로 올리기도 한다. 그래서 where e.department_id = d.department_id(+)에선 leading(d,e)를 줘도 무시되고 EMP가 선행인 HASH JOIN OUTER가 나온다(결과 107행 — 부서 없는 사원도 살아남음). 뒤집고 싶으면 swap_join_inputs(d) → HASH JOIN RIGHT OUTER(DEPT build, Buffers 13). 그리고 오늘도 오타 사례 — gather_plan_statistcs(i 누락)로 돌리자 플랜에 "basic plan statistics not available" 경고가 남고 Starts/A-Rows가 안 찍혔다. 힌트 오타는 조용히 무시되지만, 이번엔 플랜 Note가 알려줬다.

7.12Index Join — 테이블에 안 가고 인덱스끼리 조인 74일차

한 테이블에 속한 여러 인덱스를 이용해 random access 없이 결과를 만드는 방식. 질의가 참조하는 컬럼(SELECT와 WHERE 모두)이 전부 인덱스에 포함될 때만 작동한다.

rowid로 자기 자신과 조인 두 인덱스를 각각 fast full scan해서 rowid를 키로 해시 조인(access(ROWID=ROWID)) — 같은 행의 city와 location_id를 rowid로 맞춰 붙이니 테이블 블록엔 갈 필요가 없다. §5.2 "인덱스만으로 충당되면 테이블 안 간다"의 조인 버전.

filter까지 access로 — 그러나 만능은 아니다 — 기본 플랜에선 employee_id 조건이 "진입 후 확인 filter"였는데, index_join 힌트를 주자 두 조건이 각자의 인덱스에서 모두 access로 처리되고 rowid 교집합으로 끝났다(4→3). 반면 조건 없는 조회(select employee_id, job_id)에선 옵티마이저가 스스로 index join(16)을 골랐지만 full 강제가 11로 더 쌌다 — 5블록짜리 작은 테이블에선 두 인덱스 합산(4+12)이 오히려 크다. 어제부터 반복되는 결론: 판정은 실측.

8.2filter 방식 — 반복이지만, 캐시가 구한다

no_unnest 힌트로 서브쿼리를 변환 없이 filter 방식으로 돌리면(인덱스 없는 상태):

-- where department_id in (select /*+ no_unnest */ department_id from hr.dept)
| Id | Operation          | Name | Starts | A-Rows | Buffers |
|* 1 |  FILTER            |      |    1   |   106  |   47 |
|  2 |   TABLE ACCESS FULL| EMP  |    1   |   107  |   11 |
|* 3 |   TABLE ACCESS FULL| DEPT |   12   |    11  |   36 |  ← Starts 12 (107이 아니다!)
   1 - filter( IS NOT NULL)      3 - filter("DEPARTMENT_ID"=:B1)
filter operation cache (filter optimization) FILTER는 메인 루프의 행마다 서브 루프를 돌리는 구조라 NL과 실행 방식이 같다. dept full 1회가 2블록(단독 count(*) 실측 — 본 플랜의 반복 실행에선 회당 3블록으로 찍힘: 36÷12)이니 최악은 107번 × 2 = 214블록. 그런데 실측은 서브쿼리 수행이 12번뿐 — 오라클이 filter 조건의 input 값(:B1)을 cache해서, 같은 department_id가 또 오면 읽지 않고 캐시를 쓴다. emp의 department_id 종류가 12가지(null 제외 11 + 1)라 종류 수만큼만 돌았다. 술어의 :B1이 이 바인딩의 흔적이고, filter(IS NOT NULL)은 null 행 제외의 흔적이다.

8.3unnest — 서브쿼리를 조인으로 푼다 (semi join)

동일한 결과가 보장되면 옵티마이저는 서브쿼리를 조인문으로 변환한다(unnest). 조인이 되는 순간 액세스 경로·조인 방법·조인 순서를 고를 수 있게 된다 — 73~74일차의 3형제 도구가 전부 열린다. 힌트 unnest / no_unnest.

-- 같은 쿼리, 힌트 없음(인덱스 0) → 옵티마이저가 스스로 unnest
| Id | Operation           | Name | A-Rows | Buffers |
|* 1 |  HASH JOIN RIGHT SEMI|     |   106  |   12 |  ← filter 47 → semi 12
|  2 |   TABLE ACCESS FULL | DEPT |    27  |    2 |
|  3 |   TABLE ACCESS FULL | EMP  |   107  |   10 |
서브쿼리 블록에 힌트를 주는 법 — qb_name(10g~) unnest된 서브쿼리의 테이블은 메인 힌트에서 이름이 안 잡힌다. 서브쿼리에 qb_name(sub)로 이름을 붙이고, 메인에서 leading(d@sub, e)·use_nl(e)·full(d@sub)처럼 @블록이름으로 지정한다. 이걸로 unnest된 조인의 순서·방법을 3형제 실험처럼 제어했다.
주의 — 서브쿼리와 조인은 같지 않다 — 결과 건수부터 다르다(1:m에서 조인은 m쪽 만큼 늘어남). m:1처럼 결과가 보장될 때만 변환이 성립한다. 한쪽 테이블 데이터만 보면 되면 서브쿼리, 양쪽을 비교해야 하면 조인. (select distinct department_id from hr.emp)를 inline view로 조인하는 식의 우회는 곤란하다 — 함부로 unnest(수동 변환)하지 말 것.

8.4조인 제거(join elimination) — 안 읽어도 되는 테이블은 안 읽는다

1:m 관계에서 1쪽 테이블을 참조하지 않으면(SELECT에 m쪽 컬럼만) 옵티마이저가 1쪽 테이블을 아예 읽지 않는다. 전제: PK(1쪽)·FK(m쪽) 제약조건이 있어야 작동.

-- e.* 만 조회하는 employees⋈departments — DEPT를 읽지 않는다
SYS> select e.* from hr.employees e, hr.departments d
     where e.department_id = d.department_id;
-- 내부적으로 이 문장이 돌아간다:
     select * from hr.employees where department_id is not null;
-- 확인: _optimizer_join_elimination_enabled = TRUE (기본)
-- 비활성화 힌트: no_eliminate_join(d)  (수업 노트 표기 no_eliminated_join은 오타 — 잘못된 힌트는 조용히 무시됨)
FK가 있어야 하는 이유 · FK의 덤 FK가 보장하는 건 "emp의 department_id는 반드시 dept에 존재한다"는 사실 — 그래서 dept를 안 읽어도 결과가 같다고 증명이 되고, 그때만 제거한다. FK(+not null)를 걸면 ① 데이터 품질이 좋아지고 ② 조인 제거가 돌아간다. null을 포함해야 하면 outer join을 쓰되, null이 포함될 컬럼에 FK를 걸지 않는 관행도 있다(현장에선 버그로 본다). 제약조건 관련 lock 문제 — 미인덱스 FK 상태에서 부모 PK를 갱신/삭제할 때 자식(FK 쪽) 테이블이 잠기던 이슈 — 는 9i에서 완화됐다.

8.6anti join이 null을 배웠다 — ANTI NA와 LNNVL 76일차

10g까지는 NOT IN 서브쿼리가 anti join으로 풀리지 않았다 — 조인 컬럼에 is not null을 명시해야만 anti가 됐다. 지금(19c)은 힌트 없이 NOT IN을 던져도 플랜에 MERGE JOIN ANTI NA가 뜬다 — NA = Null-Aware, null의 의미(전멸)를 지키면서 anti join 구조로 도는 변환이다(그래서 결과는 여전히 0건, A-Rows 0).

-- ① is not null 명시 → 보통 anti (16행)
|* 1 |  HASH JOIN ANTI  |              | A-Rows 16 | Buffers 3 |
-- ② 명시 없이 NOT IN (19c) → null-aware anti (0행 — null 의미 유지)
|  1 |  MERGE JOIN ANTI NA |           | A-Rows  0 | Buffers 5 |
-- ③ no_unnest → filter 방식 (Buffers 57, EMP full을 Starts 27)
   1 - filter( IS NULL)   3 - filter(LNNVL("DEPARTMENT_ID"<>:B1))
LNNVL 읽는 법 LNNVL(조건) = 조건이 거짓이거나 판정 불가(null)면 참. 그래서 LNNVL("DEPARTMENT_ID"<>:B1) = department_id is null or department_id = :B1, LNNVL("DEPARTMENT_ID"=20) = department_id <> 20 or department_id is null. NOT IN의 null 의미를 술어 한 줄로 지키는 장치 — 아래 OR-expansion에서도 그대로 다시 나온다.

8.7OR-expansion — or를 union all로 다시 쓴다 76일차

job_id = 'IT_PROG' or department_id = 20 — or가 걸치면 옵티마이저는 인덱스 둘 중 하나로 못 정하고 full table scan(Buffers 7)을 잡기 쉽다. 그런데 각각은 인덱스로 4씩이다. OR-expansion = 사용자가 쿼리를 직접 바꾸지 않아도 or 조건을 union all 형태로 변환해 가지마다 index range scan을 태우는 기능.

-- use_concat 힌트 → CONCATENATION (Buffers 6)
|  1 |  CONCATENATION     |                   |
|  2 |   TABLE ACCESS ... | EMP_DEPARTMENT_IX |  ← department_id=20 가지
|* 4 |   TABLE ACCESS ... | EMP_JOB_IX        |  4 - filter(LNNVL("DEPARTMENT_ID"=20))
LNNVL이 왜 또 나오나 — 중복 제거 두 가지를 union all로 합치면 양쪽 조건을 다 만족하는 행이 두 번 나온다. 그래서 둘째 가지에 LNNVL(department_id=20)(= 20이 아니거나 null)을 자동으로 붙여 첫째 가지가 이미 가져간 행을 뺀다 — 수동으로 union all + lnnvl을 쓴 실측(6)과 use_concat(6)이 같은 플랜이 되는 이유. 힌트: use_concat(유도) / no_expand(금지, full 7).
union vs union all — union은 SORT UNIQUE(전체 중복 제거 정렬)가 하나 더 붙는다. OR-expansion이 union all + LNNVL을 쓰는 건 정렬 없이 중복을 피하는 방법이기 때문.

8.8같은 술어, 다른 플랜 — FREQUENCY 히스토그램 76일차

같은 department_id = ?인데 — 10(1행)은 index 3블록, 50(45행)은 full 9블록. 힌트 없이 옵티마이저가 값을 보고 접근 경로를 갈랐고, E-Rows도 45로 정확했다. 무엇을 보고 갈랐는지는 통계에 있다:

SYS> select column_name, num_distinct, num_buckets, histogram
     from dba_tab_columns where table_name='EMPLOYEES';
DEPARTMENT_ID   11   11  FREQUENCY   ← 값 종류 11개, 버킷 11개
JOB_ID          19   19  FREQUENCY   · LAST_NAME 102 FREQUENCY ...
히스토그램 = 값별 분포표 FREQUENCY 히스토그램은 값 종류마다 버킷 하나 — "50번 부서는 45명, 10번 부서는 1명"을 옵티마이저가 안다. 그래서 같은 술어라도 값에 따라 index/full을 바꿔 태운다. job_id = 'IT_PROG' or department_id = 50으로 OR-expansion을 걸면 job 가지는 index, 50 가지는 full — 가지마다 다른 접근(13)을 조합한다. 인덱스를 강제해도(9) full(9)과 같았다 — 45행이면 인덱스 이득이 없다. 통계·히스토그램 본편은 로드맵 09에서.

8.9pushing subquery — 서브쿼리 필터를 앞 단계로 76일차

emp⋈dept 조인에 exists (London인 location) 서브쿼리를 단 3테이블 문장. no_unnest로 filter 방식을 강제하면 서브쿼리 필터가 맨 마지막에 돈다 — MERGE JOIN으로 106행을 다 만들어 놓고 걸러낸다(Buffers 25). push_subq는 unnest 안 된 서브쿼리의 처리 순서를 제어해 가능한 앞 단계에서 먼저 필터링한다:

-- exists(select /*+ no_unnest push_subq */ ...) → dept를 읽는 단계에서 먼저 거름
|* 3 |    TABLE ACCESS FULL | DEPARTMENTS |  27→1행  3 - filter( IS NOT NULL)
|* 5 |      INDEX UNIQUE SCAN| LOC_ID_PK  |  Starts 7  5 - access("LOCATION_ID"=:B1)
|  6 |    INDEX RANGE SCAN  | EMP_DEPARTMENT_IX |  ← 1행만 NL로 (Buffers 24)
Buffers보다 구조를 봐라 25→24로 숫자는 비슷하지만 구조가 다르다 — 다음 단계로 넘어가는 행 수가 106 → 1. 데이터가 크면 이 차이가 곧 비용이다. "실행계획상 가능한 앞 단계에서 서브쿼리 필터링을 먼저 처리해 다음 수행 단계로 넘어가는 행 수를 줄인다"가 push_subq의 정의 그 자체. 참고로 힌트 없이 두면 옵티마이저는 아예 unnest해서 locations를 driving으로 3중 NL(7)을 만든다.
플랜 재구성 연습 — 큰 플랜의 Id 3·4·5만 수행하는 SQL을 직접 써보기: select /*+ full(d) */ * from hr.departments d where exists(select /*+ no_unnest */ 'x' from hr.locations where location_id = d.location_id and city='London') → FILTER + full(dept) + LOC_ID_PK(:B1). 실행계획을 보고 SQL로 되짚는 연습을 많이 하라는 게 강사 강조.

8.10view merging — 뷰를 해체해 메인과 통합한다 76일차

인라인 뷰 둘을 조인해도 플랜에 VIEW가 없다 — 옵티마이저가 뷰를 해체(merging)해서 메인 쿼리와 통합했기 때문. 뷰 없이 직접 쓴 조인과 plan hash까지 동일하다. 힌트 merge / no_merge.

구분조건·동작
simple view merging뷰 안에 group by·distinct가 없으면 해체 후 통합 — 직접 조인과 같은 플랜
complex view merginggroup by·distinct가 있어도 해체(_complex_view_merging=TRUE) — 조인 먼저, HASH GROUP BY를 꼭대기로
merging 불가뷰 안에 union·union all·intersect·minus·connect by·rownum·(group by 없는) 전체 집계·분석함수
TP(Transitive Predicate) — 상수 조건이 조인절을 타고 번진다 뷰 안의 d.department_id = 20이 조인절 e.department_id = d.department_id를 타고 emp 쪽에도 access("E"."DEPARTMENT_ID"=20)를 생성한다 — 내가 안 쓴 상수 조건이 Predicate에 보이면 TP다. 양쪽에 상수를 직접 쓴 문장과 같은 플랜(6블록)이 된다.
no_merge를 걸면 플랜에 VIEW operation이 나타나고 뷰가 독립 집합으로 처리된다(HASH JOIN). 변환을 끊고 뷰 경계를 지키고 싶을 때 쓰는 스위치이자, 아래 pushdown 실험의 준비물.

8.11조인 조건 pushdown & bloom filter — 뷰 안으로, 조인 전에 76일차

|  2 |   JOIN FILTER CREATE | :BF0000 |   |  7 |  JOIN FILTER USE | :BF0000 |

bloom filter — 조인 전에 거르는 요약 필터 hash join 시 build(선행) 집합의 조인 키를 비트 요약(:BF0000)으로 만들어 probe(후행) 쪽에 미리 배포 — probe 집합에서 조인에 참여할 가망이 없는 행을 조인 전에 걸러 건수를 줄인다. 실측에서 EMP full 107행이 JOIN FILTER USE를 거치며 2행만 올라왔다. 힌트 px_join_filter / no_px_join_filter(걸면 107행 그대로). 오늘 하루의 공통 문법이 이것 — 변환의 방향은 언제나 "더 앞 단계에서, 더 적은 행으로".

8.12조건절 pushdown & pullup — 상수 조건도 이동한다 77일차

어제(§8.11)는 조인 조건을 뷰 안으로 넣었다. 오늘은 상수 조건절 — group by 복합 뷰의 merging이 실패했을 때, 조건절 pushdown은 쿼리 블록 밖의 조건절을 안으로 넣어 group by 할 데이터양을 줄인다:

-- 뷰 밖의 where department_id = 20 이 뷰 안으로 들어갔다 (Buffers 2)
|  1 |  SORT GROUP BY NOSORT        |                   |  ← 정렬 없는 group by
|* 3 |    INDEX RANGE SCAN          | EMP_DEPARTMENT_IX |  3 - access("DEPARTMENT_ID"=20)
-- where를 뷰 안에 직접 쓴 문장과 plan hash까지 동일(2036705853) · 뷰 단독 group by는 107행/6블록
조건절 pullup — 안의 조건을 꺼내 반대편 뷰에 다시 넣는다 group by 뷰 a(=20 조건 있음)와 b(조건 없음)를 조인하면 — a 안의 department_id=20을 밖으로 끌어냈다가(pullup) b 안으로 다시 밀어 넣는다. Predicate에 5 - access(=20)·9 - access(=20)가 양쪽 뷰에 찍히고 b도 107행 group by를 면한다(Buffers 4). 이 이동 전체를 제어하는 파라미터가 _pred_move_around(기본 TRUE) — false로 끄면 b가 107행 전체를 group by(8)하고, 그 상태에서 push_pred(b)를 주면 §8.11의 조인 조건 pushdown(VIEW PUSHED PREDICATE, 4)이 대신 일한다.
SORT GROUP BY NOSORT — 인덱스(EMP_DEPARTMENT_IX)가 group by 키 순서로 값을 주니 정렬 없이 그룹 경계만 끊는다. "인덱스가 정렬을 대체한다"(§5.5 ORDER BY 흡수 → §7.9 SORT JOIN 생략)의 세 번째 재등장.

8.13집계 서브쿼리 제거 — 상호관련 집계 서브쿼리가 분석함수로 77일차

"자기 부서 평균보다 급여가 높은 사원" — e.salary > (select avg(salary) ... where department_id = d.department_id). 옵티마이저가 이 집계 상호관련 서브쿼리를 unnesting해 메인 쿼리의 분석함수로 변환한다(제어: _remove_aggr_subquery, 기본 TRUE):

|  2 |   WINDOW BUFFER|          |  ← avg() over(partition by ...)의 흔적
-- false: 서브쿼리를 group by 뷰(VW_SQ_1)로 unnest 후 조인 — Buffers 17
|  6 |     VIEW       | VW_SQ_1  |  EMP를 두 번 읽는다(107+107)
왜 분석함수 쪽이 싼가 — EMP를 한 번만 읽는다 서브쿼리 unnest 방식(VW_SQ_1)은 평균 계산용으로 EMP 전체를 한 번, 본문용으로 또 한 번 — 같은 테이블을 두 번 읽는다(17). 분석함수 변환(VW_WIF_1)은 조인 결과 위에서 avg() over(partition by department_id)를 계산하고 case ... IS NOT NULL로 거르므로 EMP를 한 번만 읽는다(8). 수동으로도 같은 문장을 만들 수 있다 — 분석함수+case를 인라인 뷰로 감싸고 밖에서 case_sal is not null(plan hash가 자동 변환과 같은 모양).

10.3bind peeking — 첫 hard parse가 값을 '훔쳐본다'

바인드 변수의 딜레마: 파싱 시점에는 값을 모른다 → 값을 모르면 히스토그램(§9.6)을 쓸 수 없다. 그래서 _optim_peek_user_binds=TRUE: 바인드 변수를 포함한 SQL이 최초로 실행될 때, 그 바인드 값을 이용해 실행계획을 수립한다. 문제는 그렇게 정해진 플랜을 이후의 모든 값이 재사용한다는 것.

히스토그램이 없을 때는 함정이 보이지 않는다. dept 10·20·50·80을 어떤 순서로 실행해도 E-Rows 10(107행÷11종 균등 가정)의 INDEX RANGE SCAN — 값이 뭐든 계산이 같으니 플랜이 갈릴 일 자체가 없다.

-- FORCE + 리터럴 dept 10→20→50→80 (히스토그램 없음) — 순서를 50 먼저로 바꿔도 동일
| 1 |  TABLE ACCESS BY INDEX ROWID BATCHED| EMP          |   10 |   690 |  2 (0)|
|*2 |   INDEX RANGE SCAN                  | EMP_DEPT_IDX |   10 |       |  1 (0)|
함정의 3재료peeking 단독으로는 무해하다. peeking + 치우친 분포 + 히스토그램이 모이면 "첫 값이 누구냐"에 따라 전체의 운명이 갈린다 — 다음 절.

10.7수동 처방 — 값 범위로 SQL을 분기한다

요구: 50·80 부서가 들어오면 full scan, 그 외 부서는 index range scan. ACS를 기다리는 대신 SQL 자체를 분기한다:

SQL> select /*+ full(e) */ * from hr.emp e
     where :b_id in (50,80) and department_id = :b_id
     union all
     select /*+ index(e emp_dept_idx) */ * from hr.emp e
     where :b_id not in (50,80) and department_id = :b_id;
-- :b_id = 50 실측: 위 브랜치만 산다
|  1 |  UNION-ALL                          |              |     1 |    45 |
|* 2 |   FILTER  filter((:B_ID=50 OR :B_ID=80))           | Starts 1 → A-Rows 45, Buffers 6
|* 3 |    TABLE ACCESS FULL                | EMP          |
|* 4 |   FILTER  filter((:B_ID<>80 AND :B_ID<>50))        | Starts 1 → 자식 Starts 0, Buffers 0
|  5 |    TABLE ACCESS BY INDEX ROWID BATCHED| EMP        |
|* 6 |     INDEX RANGE SCAN                | EMP_DEPT_IDX |

FILTER의 조건 (:B_ID=50 OR :B_ID=80)은 테이블 필터가 아니라 브랜치를 실행할지 말지 정하는 게이트다. 거짓이면 자식 Starts 0·블록 0 — 실행계획 수준에서 잠긴다. PL/SQL이면 같은 분기를 if :b_id in (50,80) then … full 힌트 … else … index 힌트로 쓰면 된다(수업의 bulk collect 예제).

이 무늬를 기억하라항상 참/거짓이 갈리는 상수 게이트 FILTER — §11.2 파티션 뷰의 NULL IS NOT NULL과 같은 원리다. "이 브랜치는 볼 필요 없음"을 옵티마이저가 플랜에 박아 넣은 것.

11.2파티션 뷰 (~7버전) — CHECK + UNION ALL로 흉내 낸 pruning

7버전까지는 파티션 테이블이 없었다 — 테이블 3개에 CHECK 제약을 걸고 UNION ALL 뷰로 묶어 파티션을 구현했다:

SQL> alter table hr.p1 add constraint c_dept_10 check(department_id < 20);
SQL> alter table hr.p2 add constraint c_dept_20 check(department_id >= 20 and department_id < 30);
SQL> alter table hr.p3 add constraint c_dept_30 check(department_id >= 30 and department_id < 40);
SQL> create or replace view hr.emp_view as
     select * from hr.p1 union all select * from hr.p2 union all select * from hr.p3;

SQL> select * from hr.emp_view where department_id = 10;
|   2 |   UNION-ALL          |          |      1 |        |      1 |       7 |
|*  3 |    TABLE ACCESS FULL | P1       |      1 |      1 |      1 |       7 |  ← 실제로 읽음
|*  4 |    FILTER filter(NULL IS NOT NULL)  |      1 |        |      0 |       0 |
|*  5 |     TABLE ACCESS FULL| P2       |      0 |      1 |      0 |       0 |  ← Starts 0
|*  6 |    FILTER filter(NULL IS NOT NULL)  |      1 |        |      0 |       0 |
|*  7 |     TABLE ACCESS FULL| P3       |      0 |      1 |      0 |       0 |  ← Starts 0

CHECK 제약이 "P2·P3에는 dept=10이 있을 수 없다"를 보증하므로, 옵티마이저는 그 브랜치 위에 항상 거짓인 게이트 filter(NULL IS NOT NULL)를 심는다 → 자식 Starts 0·Buffers 0. dept=10이면 7블록, 조건이 없으면 3테이블 전부 18블록.

왜 NULL IS NOT NULL인가옵티마이저가 "이 브랜치는 결과가 없다"를 표현하는 관용구다 — 항상 거짓이므로 자식을 시작조차 안 한다. 근거는 데이터가 아니라 CHECK 제약이라는 약속: 제약이 없으면 세 테이블을 다 읽어야 한다. §10.7 분기 union all의 :B_ID=50 OR :B_ID=80 게이트와 같은 무늬 — 저쪽은 바인드 값이, 이쪽은 제약이 게이트의 진릿값을 정한다.

11.3range 파티션 (8버전) — 이력 데이터의 기본형

  • 이력성 데이터 조회에 유리 — 기간 조건이 파티션 키에 떨어진다.
  • 여러 컬럼을 파티션 키로 지정 가능(최대 16개).
  • maxvalue 파티션으로 경계 밖 값을 받는다.
SQL> create table hr.emp_year
     partition by range(hire_date)
     ( partition p2004 values less than(to_date('2005-01-01','yyyy-mm-dd')),
       partition p2005 values less than(to_date('2006-01-01','yyyy-mm-dd')),
       partition p2006 values less than(to_date('2007-01-01','yyyy-mm-dd')),
       partition pmax  values less than(maxvalue) )
     as select employee_id, last_name, salary, hire_date, department_id from hr.employees;
-- less than(2005-01-01) = "< 2005-01-01(미만)" · pmax = ">= 2007-01-01 전부"
SQL> select partition_name, high_value from dba_tab_partitions where table_name='EMP_YEAR';
P2004  TO_DATE(' 2005-01-01 00:00:00', 'SYYYY-MM-DD...')   ← 경계는 딕셔너리에
P2005  TO_DATE(' 2006-01-01 ...)   P2006  TO_DATE(' 2007-01-01 ...)   PMAX  MAXVALUE
SQL> select * from hr.emp_year partition(p2004);   -- 파티션 지정 조회
SQL> select column_name from dba_part_key_columns where name='EMP_YEAR';   → HIRE_DATE
왜 range가 기본형인가보관주기 관리와 조건이 같은 축(기간)이기 때문이다. "2004년 데이터"는 조회 조건이면서 동시에 백업·삭제의 단위 — pruning(성능)과 파티션 drop(관리)이 한 설계에서 나온다. 경계 정의는 딕셔너리(high_value)가 문서 역할을 하므로, 파티션 구성이 궁금하면 DDL이 아니라 dba_tab_partitions를 본다.

11.4interval 파티션 (11g) — 정해진 간격으로 자동 증설

range는 경계를 사람이 미리 만들어야 한다. interval 기준으로 정의하면 정해진 간격으로 파티션이 자동 추가된다:

SQL> create table hr.emp_year
     partition by range(hire_date) interval(numtoyminterval(1,'year'))
     ( partition p2004 values less than(to_date('2005-01-01','yyyy-mm-dd')),
       partition p2005 values less than(to_date('2006-01-01','yyyy-mm-dd')),
       partition p2006 values less than(to_date('2007-01-01','yyyy-mm-dd')) ) ...
-- 2007 이후 데이터가 들어오자 1년 간격 파티션이 자동 생성:
SQL> select * from hr.emp_year partition(SYS_P769);   -- SYS_P770 …

maxvalue 파티션 없이도 넘치는 값이 자동 수용된다는 점이 range와의 차이. 자동 생성 파티션 이름은 SYS_P<n>. 간격 정의에 쓰는 interval 함수들(78일차 곁가지의 연장):

함수뜻실측
to_yminterval('1-10')1년 10개월 (interval year to month 리터럴)sysdate + → 10-MAY-28
numtoyminterval(1,'year')숫자 → year/month intervalinterval 파티션 간격 정의에 사용
to_dsinterval('10 10:10:10')10일 10:10:10 (day to second 리터럴)localtimestamp + → 21-JUL-26 02:29:58
numtodsinterval(10,'day')숫자 → day/hour/minute/second interval'hour'·'minute'·'second'도 동일
왜 interval이 필요한가range의 약점은 미래 경계를 사람이 계속 만들어야 한다는 것 — 연말마다 파티션 추가 작업을 잊으면 maxvalue에 데이터가 쌓이거나 insert가 실패한다. interval은 "간격"만 약속하면 경계 밖 첫 데이터가 도착하는 순간 오라클이 파티션을 만들어 준다(실측 SYS_P769·770). 운영 실수 한 종류가 설계로 사라지는 셈.

11.5hash 파티션 (8i) — 의미가 아니라 양으로 나눈다

  • 파티션 키 값에 해시 함수를 적용해 분할 — 경계(high_value)가 없다.
  • 중복이 많은 값보다 유일한 키·분포가 고른 컬럼을 파티션 키로 선정한다.
  • 파티션 개수는 2의 제곱(2, 4, 8, 16, …)으로.
  • pruning은 =·in 조건에서만 동작한다 — 해시라 범위 조건은 못 자른다.
  • 병렬 쿼리 성능뿐 아니라 동시 입력이 많은 테이블·인덱스의 경합을 줄일 목적으로 쓴다.
SQL> create table hr.emp_hash partition by hash(employee_id) partitions 4 ...
SQL> select partition_name, high_value from dba_tab_partitions where table_name='EMP_HASH';
SYS_P771 (high_value 없음)  SYS_P772  SYS_P773  SYS_P774
range/list vs hashrange·list는 데이터의 의미(기간·그룹)로 나눠 검색 범위를 줄이는 게 목적이고, hash는 양으로 균등하게 흩어 넣을 때의 경합(같은 세그먼트로 몰림)을 줄이는 게 목적이다.

11.6list 파티션 (9i) — 정해진 그룹, 그리고 default

  • 미리 정해진 값 목록을 기준으로 분할 저장.
  • 파티션 키는 단일 컬럼만 지정 가능.
SQL> create table hr.emp_list partition by list(department_id)
     ( partition p_dept_1 values(10,20,30,40),
       partition p_dept_2 values(50),
       partition p_dept_3 values(60,70,80,90,100,110),
       partition p_dept_4 values(default) ) ...
SQL> select * from hr.emp_list partition(p_dept_4);
178  Grant   7000  24-MAY-07   (DEPARTMENT_ID null)  ← 목록 밖 값·NULL은 default로

11.7composite 조합 파티션 — range-hash 4×4와 Pstart/Pstop 읽기

주 파티션 키로 1차 분배하고, 서브 파티션 키가 최종 저장 위치(세그먼트)를 결정한다 — 세그먼트는 서브 파티션마다 하나씩. 조합 연혁: range-hash(8i) · range-list(9i) · range-range·list-hash·list-list·list-range(11g).

SQL> create table hr.emp_comp
     partition by range(salary) subpartition by hash(employee_id) subpartitions 4
     ( partition p4999 values less than(5000), partition p9999 values less than(10000),
       partition p19999 values less than(20000), partition pmax values less than(maxvalue) ) ...
-- 4 파티션 × 4 서브파티션 = 세그먼트 16개 (SYS_SUBP1084~1099):
--   p4999  [h1 h2 h3 h4]     p9999  [h5 h6 h7 h8]
--   p19999 [h9 h10 h11 h12]  pmax   [h13 h14 h15 h16]  ← 서브파티션은 전체 일련번호

display_cursor(...,'allstats last +partition')으로 Pstart/Pstop을 보면, 어떤 조건이 몇 개 세그먼트를 읽는지가 그대로 드러난다:

조건RANGE 연산HASH 연산읽은 세그먼트Buffers
salary<5000 and emp_id=200SINGLE (1·1)SINGLE (1·1)1/163
salary<5000 and emp_id=105SINGLE (1·1)SINGLE (3·3)1/164
salary<5000 (범위 키만)SINGLE (1·1)ALL (1–4), Starts 44/1612
salary>20000SINGLE (4·4)ALL (13–16)4/162
emp_id=100 (해시 키만)ALL (1–4)SINGLE (3·3), Starts 44/168
salary>20000 and emp_id=100SINGLE (4·4)SINGLE (15·15)1/162
-- 두 키를 다 주면 16분의 1만 읽는다:
|  1 |  PARTITION RANGE SINGLE|          | Pstart 4  Pstop 4  |
|  2 |   PARTITION HASH SINGLE|          | Pstart 15 Pstop 15 |  ← pmax의 3번째 = 전체 15번
|* 3 |    TABLE ACCESS FULL   | EMP_COMP | Buffers 2 |

키 하나만 확정되면 그 축은 SINGLE, 못 정한 축은 ALL(Starts가 그 개수만큼 반복). HASH의 Pstart/Pstop이 13–16, 15처럼 나오는 것은 서브 파티션의 전체 일련번호다(4번 파티션의 h1~h4 = 13~16).

pruning도 산수다E-Rows가 산수였듯(§9.4), pruning은 "조건이 파티션 키를 확정하는가"의 문제다. 조건에 어느 키를 넣느냐가 읽을 세그먼트 수(1/16 vs 4/16 vs 16/16)를 정한다 — where절 설계가 곧 I/O 설계.

11.8정적 vs 동적 pruning — 언제 아느냐의 차이 80일차

partition pruning = SQL 조건절을 분석하여 읽지 않아도 되는 파티션 세그먼트를 액세스 대상에서 제외시키는 기능. 실습은 535,000행 sal_emp를 range(salary) 4파티션(p1<5000 · p2<10000 · p3<30000 · pmax)으로 재생성해 진행.

왜 구분해 두나바인드 변수면 값에 따라 액세스할 파티션이 execute마다 달라진다 — 플랜만 보고 "어느 파티션을 읽는다"고 단정할 수 없고 KEY로 읽어야 한다. 지난 §11.7 실측이 전부 상수 조건(정적)이었다는 것도 함께 기억할 것. (KEY 표기의 실제 캡처는 없음 — 홈랩 재현 목록)

11.9파티션 관리 ① — split · add(ORA-14074) · rename · drop · truncate 80일차

파티션 관리는 업무팀 요청이 말로 들어오면 바로 명령어로 수행해야 하는 영역 — 따로 문서화해 두고 외울 것(강사 강조).

-- split: 나누기 (p3 <25000 을 20000 경계로 둘로)
SQL> alter table hr.sal_emp split partition p3 at(20000) into (partition p3, partition p4);
-- add: 마지막 파티션보다 큰 경계만 가능
SQL> alter table hr.sal_emp add partition p5_1 values less than(30000);   -- OK
SQL> alter table hr.sal_emp add partition p2_1 values less than(10000);
ORA-14074: partition bound must collate higher than that of the last partition
-- 기존 high_value보다 작은 값은 add 불가 → split을 이용해야 한다
SQL> alter table hr.sal_emp rename partition p5_1 to p5;   -- 통계는 사라지지 않는다
SQL> alter table hr.sal_emp drop partition p5;
SQL> alter table hr.sal_emp truncate partition p3;   -- p3만 비움 → no rows selected
  • rename의 용도 — interval 파티션 생성 시 오라클이 임의로 만든 이름(SYS_P…, §11.4)을 의미 있는 이름으로 정리.
  • drop 후 복구하려면? — timebased recovery. 언제 지웠는지는 로그마이너 분석으로 찾는다(supplemental log data가 어떻게 되어 있는지부터, 일련의 과정을 깔끔하게 설명할 수 있어야 — 운영편과 연결).
add가 막히는 이유range 파티션의 경계는 "미만(less than)"의 누적 사다리다 — 중간에 새 칸을 끼우면 기존 칸의 의미(어디까지 담겼나)가 깨진다. 그래서 끝에 더하는 것(add)만 허용하고, 중간은 기존 칸을 쪼개는 것(split)으로만 만든다.

11.10파티션 관리 ② — row movement(ORA-14402) · maxvalue(ORA-14400) · merge · exchange 80일차

-- 파티션 키 update: 같은 파티션 안이면 OK, 경계를 넘으면 에러
SQL> update hr.sal_emp set salary = 18000 where employee_id = 101;   -- p3 안 → 1 row updated
SQL> update hr.sal_emp set salary = 21000 where employee_id = 101;   -- p3 → p4로 이동해야
ORA-14402: updating partition key column would cause a partition change
SQL> alter table hr.sal_emp enable row movement;
SQL> update hr.sal_emp set salary = 21000 where employee_id = 101;   1 row updated.
SQL> alter table hr.sal_emp disable row movement;   -- 작업 후 꼭 disable 부탁하기

-- maxvalue 파티션이 없을 때 경계 밖 insert
SQL> insert into hr.sal_emp values(300,'oracle',29000,20);
ORA-14400: inserted partition key does not map to any partition
SQL> alter table hr.sal_emp add partition pmax values less than(maxvalue);   -- 이제 수용
-- pmax에 30000 경계를 만들고 싶다? add 안 됨(14074) → split
SQL> alter table hr.sal_emp split partition pmax at(30000) into (partition p5, partition pmax);

-- merge: 합치기 — 문법 주의(partitions 복수형)
SQL> alter table hr.sal_emp merge partitions p4,p5 into partition p5;
-- merge하는 순간, 없었던 통계가 자동으로 수집된 것을 확인할 수 있다

exchange — 일반 테이블과 파티션의 세그먼트를 통째로 교체한다. insert 서브쿼리로 옮기고 지우는 대신 이름표만 바꿔 다는 것:

  • 업무팀이 "일반 테이블처럼 load 시켜달라" 하면 일반 테이블에 넣고 exchange — insert 서브쿼리보다 훨씬 편리.
  • 빈 테이블과 exchange하면 truncate 대용으로도 쓸 수 있다(오라클의 노하우).
  • alter 작업은 DBA가 하는 일이다 — truncate도 마찬가지.

11.11파티션의 공간·통계 — 1행인데 1,006블록, shrink로 되돌리기 80일차

-- truncate·exchange·DML을 반복한 뒤 파티션별 통계:
SQL> exec dbms_stats.gather_table_stats('hr','sal_emp',granularity=>'auto')
POSITION PARTITION HIGH_VALUE TABLESPACE NUM_ROWS BLOCKS AVG_ROW_LEN
3        P3        20000      USERS      1        1006   19   ← 1행인데 1,006블록!
-- "오라클에서는 버그라고 말하지 않는다. (버그 냄새가 난다)" — 어떻게 하나? shrink.
SQL> alter table hr.sal_emp enable row movement;   -- 행 이동 발생 → 필수
SQL> alter table hr.sal_emp modify partition p3 shrink space cascade;
SQL> alter table hr.sal_emp disable row movement;
-- 특정 파티션만 통계 수집: 파티션 이름은 문자열 취급
SQL> exec dbms_stats.gather_table_stats('hr','sal_emp',partname=>'p3', granularity=>'partition')
왜 이런 일이 생기나파티션도 세그먼트라 HWM 아래 공간은 반납되지 않는다(내부구조편) — 행이 1개 남아도 블록 사용량은 과거의 최대치를 기억한다. full scan류는 HWM까지 읽으므로 이대로 두면 "1행 읽는데 1,006블록"이 계속된다. 재구성(shrink)은 행을 옮기므로 enable row movement가 선행 — §11.10의 ORA-14402와 같은 이유다.

11.12shrink 검증 — 1,006블록이 1블록으로, 그리고 8MB의 비밀 81일차

지난 시간 홈랩 숙제였던 shrink의 효과가 실측으로 확인됐다:

SQL> alter table hr.sal_emp enable row movement;
SQL> alter table hr.sal_emp modify partition p3 shrink space cascade;
SQL> alter table hr.sal_emp disable row movement;
SQL> exec dbms_stats.gather_table_stats('hr','sal_emp',partname=>'p3',granularity=>'partition')
P3   NUM_ROWS 1   BLOCKS 1   ← 1,006 → 1 (이후 조회에선 6으로 보임 — 시점 차)
SQL> select * from hr.sal_emp partition(p3);   -- allstats: RANGE SINGLE, Buffers 2

같은 딕셔너리 조회에 initial_extent를 붙여 보면 모든 파티션이 8,388,608(8MB)로 시작한다 — 히든 파라미터 _partition_large_extents=TRUE(기본, x$ksppcv로 확인)가 파티션 세그먼트의 초기 익스텐트를 8MB로 잡기 때문. alter session set "_partition_large_extents"=false로 끌 수 있다(수업은 값 확인까지).

작은 파티션이 많다면행 몇 개짜리 파티션도 8MB를 먼저 받는다 — 파티션을 잘게 많이 쪼갤수록 "쓰지도 않는 8MB"가 쌓인다. 어제의 1,006블록(HWM)과는 다른 결의 공간 이슈 — 하나는 과거의 기억, 하나는 시작 크기다.

11.13Data Pump와 파티션 — 조각 단위로 백업하고 복구한다 81일차

파티션의 관리 이점이 실전이 되는 순간 — Data Pump(expdp/impdp)는 파티션 단위로 export/import할 수 있다. 먼저 디렉토리 객체:

부분범위처리 = 조건을 만족하는 데이터를 전체 범위로 처리하지 않고 운반단위(arraysize)까지만 먼저 처리해 유저 프로세스에 전달하고, 다음 작업을 계속하겠다는 요구가 있을 때까지 잠정적으로 수행을 멈추는 처리 방식. 예: 10,000건을 추출할 때 운반단위가 1,000건이면 한꺼번에 다 읽지 않고 1,000개씩 10번으로 나누어 처리할 수 있다.

12.1부분범위가 안 되는 SQL — 그리고 대체 방안

그룹함수·group by·sort — 집계값은 다 읽어야 나온다—

union · minus · intersectunion → union all + not exists · minus → not exists · intersect → exists

대체가 어려우면 arraysize를 넉넉하게 해서 수행한다. 웹페이지는 전부 부분범위(page) 처리이고, SQL Developer가 한 번에 50개씩 가져오는 것도 같은 원리(50이 arraysize).

왜 union all인가union·minus·intersect는 중복 제거·집합 연산을 위해 전체를 모아 정렬/비교해야 하지만, union all은 오는 대로 흘려보내면 된다 — §8.7 OR-expansion, §10.7 분기 SQL이 전부 union all이었던 것도 같은 맥락.

12.3arraysize × 파티션 — 함께 봐야 하는 이유

수업 후반 실험(캡처 없음 — 홈랩 재현 목록): salary between 5000 and 8000을 arraysize 15/100/1000/2000으로 바꿔가며, 그리고 인덱스 생성·파티셔닝 후에 각각 allstats last +partition으로 Buffers를 비교했다. 강사 관찰 요지:

  • 대용량을 full scan하는 것도 부담, index range scan은 더 부담(random I/O) — 이럴 때 파티셔닝. 나름 I/O를 줄일 수 있다 — 이유를 알고 파티셔닝을 해야 한다.
  • 파티셔닝이 되어 있더라도 arraysize가 15면 I/O(gets)가 증가한다 — fetch 재방문은 파티션이 해결해 주지 않는다.
  • arraysize 100 이상에서는 파티셔닝 전보다 I/O 수가 줄었다.
두 다이얼은 서로 다른 낭비를 줄인다파티션(pruning)은 어느 세그먼트를 읽나를 줄이고, arraysize는 같은 블록을 몇 번 다시 집나를 줄인다. 하나만 돌리면 다른 쪽 낭비가 남는다 — §12.2의 임계점과 §11.7의 SINGLE/ALL을 한 화면에 놓고 볼 것.

13.2로컬 파티션 인덱스 — 테이블 조각과 1:1, 140 → 3

  • 파티션 테이블의 각 파티션과 파티션 인덱스의 각 파티션이 1대1로 매핑되는 인덱스.
  • 테이블 파티션 개수 = 인덱스 파티션 개수, 테이블 파티션 키 = 인덱스 파티션 키.
  • 유지 관리 작업은 오라클이 자동으로 관리한다 — 다른 파티션에 영향을 주지 않는다.
  • PREFIXED = 파티션 키 컬럼을 인덱스 키 컬럼의 왼쪽 서두에 두는 것.
SQL> create unique index hr.emp_local_idx on hr.emp_local(employee_id) local;

SQL> select * from hr.emp_local where employee_id = 1000;

|* 3 |    INDEX UNIQUE SCAN               | EMP_LOCAL_IDX |

13.4글로벌 파티션 인덱스 — 테이블과 다른 축으로 자른다

정의(원문): 파티션 테이블의 파티션 키와 파티션 인덱스의 파티션 키가 일치하지 않는다. 테이블은 employee_id로, 인덱스는 hire_date로:

SQL> create index hr.emp_global_idx on hr.emp_global(hire_date) global
     partition by range(hire_date)

EMP_GLOBAL_IDX  NONUNIQUE  GLOBAL  PREFIXED  YES  PARTITION_COUNT 4

수동 관리는 세션까지만. workarea_size_policy=manual + sort_area_size 등을 직접 잡을 수 있지만 시스템 레벨에선 절대 금지(전체 세션이 물림). 실험은 세션에서만.

15.3정렬의 세 등급 — optimal · onepass · multipass

정렬이 필요하면 오라클은 PGA의 SORT AREA에서 수행한다. 다 담기면 메모리 소트(internal), 넘치면 정렬 중간 결과(sort run)를 temp 세그먼트에 흘리는 디스크 소트(external):

등급뜻디스크 I/O
optimal정렬이 SORT_AREA_SIZE 안에서 완료없음 (최적)
onepasssort run이 디스크에 한 번만 쓰임1회
multipasssort run이 디스크에 여러 번 쓰임많음 → 성능 저하
-- sort_area_size를 줄여 가며 emp_temp 정렬 → Used-Temp 관찰
SQL> alter session set workarea_size_policy = manual;
SQL> alter session set sort_area_size = 1048576;   -- 1MB
SQL> select … order by employee_id …;   -- allstats last에 Used-Tmp 등장
# Used-Temp: 메모리가 부족해 temp 공간을 쓴 크기(KB)
SQL> -- v$session_event: 42 direct path read temp / 42 direct path write temp
child cursor가 갈리는 또 하나의 이유같은 SQL이라도 sort_area_size처럼 옵티마이저 환경이 다르면 child cursor가 새로 생긴다(§10.1의 커서 공유). v$sql_shared_cursor로 갈린 이유를, v$sql_workarea의 last_tempseg_size로 디스크 사용을 추적한다 — 디스크 소트가 많을수록 direct path read/write temp 대기가 쌓인다.

15.4정렬이냐 해시냐 — 결과가 정렬돼 나오는지로 읽는다

연산오퍼레이션결과 정렬?
order by · 집계(sum) 준비SORT (ORDER BY · AGGREGATE)예 — SORT AREA에서
group by · distinct · uniqueHASH (GROUP BY · UNIQUE)아니오 — HASH AREA에서

9iR2부터 _gby_hash_aggregation_enabled=TRUE(기본) — group by·distinct·unique를 해시로 수행한다. 그래서 HASH GROUP BY·HASH UNIQUE가 뜨고 결과가 정렬돼 나오지 않는다. 알고리즘이 바뀐 것:

SQL> select distinct department_id from hr.employees;   → HASH UNIQUE (정렬 안 됨)
-- 옛 방식(정렬된 결과)을 보려면:
SQL> select /*+ opt_param('_gby_hash_aggregation_enabled','false') */ distinct department_id …;
→ SORT UNIQUE (정렬됨)
정렬이 필요없는 집계는 해시가 이긴다group by·distinct는 "중복을 뭉치는" 일이지 "순서를 매기는" 일이 아니다 — 그래서 정렬(O(n log n))보다 해시(O(n))가 유리하다. 정렬된 결과가 필요하면 명시적 order by를 붙이거나(그럼 SORT가 추가) 힌트로 옛 알고리즘을 부른다. 정렬 결과를 만드는 오퍼레이션(SORT ORDER BY·SORT GROUP BY·SORT UNIQUE·MERGE JOIN의 SORT)을 구분해 읽는 게 실행계획 독해의 마지막 조각. §15가 PGA(프로세스 전용 작업 공간)였다면, 여기는 SGA의 버퍼 캐시를 어떻게 나눠 쓰고, 그 모든 메모리를 어떻게 자동으로 관리하는가다. 튜닝의 마지막 무대는 "무엇을 메모리에 남길 것인가".

16.1Database Buffer Cache — 그리고 세 개의 풀

Data Buffer Cache = 데이터파일에서 읽은 블록의 복사본을 담는 메모리 — 물리 I/O를 최소화하려 최근 블록을 남긴다(모든 유저가 공유·동적 크기 조정). 기본은 DEFAULT 풀에 담기지만, 성격이 다른 데이터를 위해 풀을 나눌 수 있다:

풀어떤 데이터목적 · 파라미터
DEFAULT일반기본 — db_cache_size
KEEP사용 빈도 높고 작은 테이블재사용 블록을 캐시에 붙잡아 둠 — db_keep_cache_size
RECYCLE빈도 낮고 읽는 양 많은 테이블재사용 안 될 블록을 격리해 DEFAULT를 보호 — db_recycle_cache_size
nK표준 블록과 다른 크기nonstandard 블록 테이블스페이스용 — db_{2,4,8,16,32}k_cache_size
KEEP과 RECYCLE은 정반대 목적KEEP은 "자주 쓰는 작은 것을 안 밀려나게" — 코드 테이블처럼. RECYCLE은 "한 번 크게 훑고 버릴 것이 남의 캐시를 밀어내지 않게" — 배치 대상 대용량처럼. 둘 다 DEFAULT 풀의 LRU를 지키는 장치다(내부구조편의 LRU·touch count와 이어진다).

16.2풀 지정 실습 — 세그먼트에 꼬리표를 단다

SQL> alter system set db_keep_cache_size = 16m scope=both;   -- 풀부터 만들고
SQL> select name, block_size, buffers, current_size from v$buffer_pool;
KEEP     8192  1958   16      DEFAULT  8192  35244  288   ← DEFAULT에서 떼어 KEEP에 할당
SQL> alter table hr.employees storage(buffer_pool keep);   -- 세그먼트에 꼬리표
SQL> alter index hr.emp_emp_id_pk storage(buffer_pool keep);    -- 인덱스도 개별 지정
SQL> alter table hr.employees storage(buffer_pool default);    -- 되돌리기

블록 크기가 다른 nonstandard 블록 테이블스페이스를 쓰려면, 먼저 그 크기의 캐시를 열어야 한다 — 표준(db_block_size 8K)과 별개다:

SQL> alter system set db_4k_cache_size = 16m;   -- 4K 캐시부터
SQL> create tablespace oltp_tbs datafile '…oltp_tbs01.dbf' size 5m blocksize 4k
     extent management local uniform size 1m segment space management auto;
SQL> create table hr.oltp_emp tablespace oltp_tbs as select * from hr.employees;
-- 같은 employees가 8K 블록엔 8블록, 4K 블록 세그먼트엔 더 잘게 나뉜다(블록당 용량 절반)
세그먼트마다 캐시가 달라질 수 있다. 테이블·인덱스별로 buffer_pool 꼬리표를 달고(dba_tables·dba_indexes의 buffer_pool), 블록 크기가 다르면 그에 맞는 nK 캐시가 필요하다 — "이 데이터는 어느 메모리에 살까"를 설계로 정하는 것.

16.3Redo Log Buffer — 목적은 오직 복구

  • DB 데이터 블록의 모든 변경사항을 기록한다 — 기본 목적은 복구(recovery). 기록 단위가 redo entry(변경을 재구성·재실행할 정보).
  • redo가 생성되는 SQL — DML(insert·update·delete·merge) · DDL(create·alter·drop) · select … for update(잠금도 변경).
  • log_buffer로 크기 조정 — static parameter라 scope=spfile로만 바꾸고 재기동해야 반영.
순수 select은 왜 빠지나redo는 "무엇을 되돌리고 다시 하나"의 기록이라 변경이 있어야 생긴다 — 읽기만 하는 select은 바꾼 게 없으니 redo가 없다. 예외가 select for update: 행에 잠금을 걸어 상태를 바꾸므로 redo가 남는다.

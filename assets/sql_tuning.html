SQL Tuning

구조를 이해했다면, 이제 그 구조를 활용해 같은 결과를 더 빠르게 얻는 일. 내부구조편(완전정복 가이드)이 "어떻게 동작하나"라면, 여기는 "어떻게 빠르게 하나"다. 액세스 경로부터 시작해 조인·힌트·실행계획으로 쌓아간다.

01
SQL Tuning이란
시작은 SQL의 의미를 파악하는 것

SQL Tuning은 SQL문을 최적화하여 빠른 시간 안에 원하는 결과값을 얻기 위한 작업이다. 그 출발점은 도구나 힌트가 아니라 — SQL의 의미(작성 의도)를 제대로 파악하는 것이다. "이 쿼리가 진짜 원하는 게 무엇인가, 얼마나 많은 데이터를 건드려야 하는가"를 알아야 올바른 처리 방법을 고를 수 있다.

전제 — 튜닝은 내부구조편의 지식 위에서 동작한다. 블록·HWM·버퍼 캐시·인덱스 구조를 알아야 "왜 이 실행계획이 느린가"가 보인다. 여기서는 그 지식을 활용하는 쪽에 집중한다.

02
실행계획 읽기 — 모든 튜닝의 기반
계획을 뽑고, 순서대로 읽고, 느린 지점을 찾는다

튜닝의 첫 동작은 인덱스를 만드는 게 아니라 실행계획을 읽는 것이다. "옵티마이저가 데이터를 어떤 경로로·어떤 순서로 가져오는가"를 못 읽으면 무엇을 고칠지 알 수 없다. 이 장이 안 되면 뒤의 모든 튜닝이 공허하다.

2.1실행계획 뽑는 3가지 방법

방법명령성격
autotraceset autotrace traceonly explain예상 계획만 (실행 안 함)
EXPLAIN PLANexplain plan for …; → select * from table(dbms_xplan.display);예상 계획 (실행 안 함)
display_cursor/*+ gather_plan_statistics */ 실행 후 → dbms_xplan.display_cursor(format=>'ALLSTATS LAST')실제 실행 계획 + 실측치
예상 vs 실제 — 가장 중요한 구분 explain plan·autotrace explain은 옵티마이저의 예상(E-Rows)만 보여준다. 반면 display_cursor는 실제로 돌린 뒤 A-Rows(실제 행 수)까지 보여준다. E-Rows와 A-Rows가 크게 차이 나면 옵티마이저가 데이터를 잘못 추정한 것 — 잘못된 계획의 가장 흔한 원인이다. 그래서 진짜 튜닝은 display_cursor로 한다.

2.2읽는 순서 — 안쪽·위부터

실행계획은 트리다. 들여쓰기가 가장 깊은(안쪽) 라인부터, 같은 깊이면 위에서 아래로 실행되고, 자식이 만든 결과를 부모에게 올린다.

2.3핵심 컬럼 읽는 법

컬럼의미 · 볼 점
Id단계 번호. *가 붙으면 아래 Predicate Information에 조건이 있다는 표시
Operation무슨 작업인가 (FULL/INDEX/JOIN…). 들여쓰기로 부모-자식
Name대상 객체(테이블·인덱스 이름)
Rows옵티마이저가 예상한 행 수(카디널리티). 실제와 괴리가 크면 위험 신호
Cost (%CPU)옵티마이저의 상대적 비용 추정. 절대 시간이 아니라 비교용
Predicate (access/filter)access=인덱스로 범위를 좁힘(좋음) / filter=읽은 뒤 걸러냄(많이 읽었을 수)
access vs filter — 같은 조건이라도 access("ID"=100)로 잡히면 인덱스로 바로 찾은 것이고, filter("ID"=100)로 잡히면 일단 많이 읽고 나서 버린 것이다. filter로만 처리되는 핵심 조건이 보이면 인덱스 기회다.

2.4자주 보는 Operation 사전

Operation한마디로
TABLE ACCESS FULL테이블 전체 스캔(첫 블록~HWM, scattered read)
TABLE ACCESS BY INDEX ROWID인덱스가 준 rowid로 그 행만 읽기
INDEX UNIQUE SCAN유일 인덱스로 정확히 1행
INDEX RANGE SCAN범위 조건으로 인덱스 일부 스캔
INDEX FULL / FAST FULL SCAN인덱스 전체(정렬 활용 / multi-block)
INDEX SKIP SCAN선두 컬럼을 건너뛰고 결합 인덱스 활용
NESTED LOOPS한쪽 행마다 다른 쪽을 반복 조회(소량 조인)
HASH JOIN해시 테이블을 만들어 대량 조인
SORT (ORDER BY/GROUP BY/UNIQUE)정렬·중복 제거
FILTER조건·서브쿼리로 걸러내기

2.5예제로 읽어보기

2.2의 계획을 실제로 찍으면 이렇게 나온다. display_cursor로 E-Rows(예상)와 A-Rows(실제)를 같이 보는 게 핵심이다.

SYS@ora19c> select /*+ gather_plan_statistics */ e.last_name, d.department_name
          from hr.employees e join hr.departments d using(department_id)
          where e.department_id = 50;
SYS@ora19c> select * from table(dbms_xplan.display_cursor(format=>'ALLSTATS LAST'));

----------------------------------------------------------------------
| Id | Operation                     | Name        | E-Rows | A-Rows |
----------------------------------------------------------------------
|  0 | SELECT STATEMENT              |             |        |     45 |
|  1 |  NESTED LOOPS                 |             |     45 |     45 |
|* 2 |   TABLE ACCESS BY INDEX ROWID | EMPLOYEES   |     45 |     45 |
|* 3 |    INDEX RANGE SCAN           | EMP_DEPT_IX |     45 |     45 |
|  4 |   TABLE ACCESS BY INDEX ROWID | DEPARTMENTS |      1 |      1 |
|* 5 |    INDEX UNIQUE SCAN          | DEPT_ID_PK  |      1 |      1 |
----------------------------------------------------------------------
Predicate Information:
   3 - access("E"."DEPARTMENT_ID"=50)
   5 - access("E"."DEPARTMENT_ID"="D"."DEPARTMENT_ID")
  • E-Rows = A-Rows (45 = 45) → 옵티마이저 추정이 정확하다. 괴리가 없으니 통계는 건강하다.
  • 핵심 조건이 access(... =50) → 인덱스로 바로 범위를 좁혔다(좋음). filter가 아니다.
  • 50번 부서 45명만 인덱스로 집어 각자 부서를 unique scan으로 붙였다 — 소량이라 NESTED LOOPS가 적합.
무엇을 의심하나 (체크 순서) ① 큰 테이블에 TABLE ACCESS FULL인데 소량만 찾나? → 인덱스 기회. ② 핵심 조건이 filter로만 처리되나? → 인덱스 기회. ③ E-Rows ≪ A-Rows로 추정이 크게 빗나갔나? → 통계 갱신·카디널리티 문제. ④ 소량인데 HASH JOIN, 대량인데 NESTED LOOPS? → 조인 방식 의심.
홈랩에서 바로 — ora19c에서 set autotrace on 또는 위 gather_plan_statistics + display_cursor로 직접 찍어보세요. 자료를 읽는 것보다 내 쿼리의 계획을 눈으로 보는 게 훨씬 빨리 늡니다.
03
데이터 액세스 경로
무엇을 어떻게 읽을지 — 튜닝의 핵심 갈림길

옵티마이저가 데이터를 가져오는 방법은 크게 둘이다. 어느 쪽이 빠른지는 "얼마나 많은 행을 읽느냐"에 달렸다.

경로설명유리한 경우
full table scan세그먼트 첫 블록부터 HWM까지 통째로 읽음많은 데이터 검색
rowid scan · by user rowid사용자가 rowid를 직접 지정해 한 행을 콕 집음rowid를 이미 알 때
rowid scan · by index rowid인덱스를 타고 rowid를 얻어 해당 블록의 행을 읽음소량 검색
직관 · full scan vs index의 분기점 전체의 작은 일부만 찾을 땐 인덱스(by index rowid)가 빠르다. 하지만 전체의 상당 비율을 읽어야 한다면, 인덱스를 타며 블록을 하나씩 random하게 읽는 것보다 full scan으로 multi-block I/O로 쓸어 담는 게 오히려 빠르다. "인덱스가 항상 빠르다"는 흔한 오해다 — 읽어야 할 양이 기준이다.
04
Full Table Scan
첫 블록부터 HWM까지, multi-block로

full table scan은 세그먼트의 첫 블록부터 HWM(High Water Mark, 사용 블록과 미사용 블록의 경계)까지 읽는다. 핵심은 한 번의 I/O call에 db_file_multiblock_read_count(이 환경 =78)만큼의 블록을 묶어 읽는 Multi Block I/O라는 점. 그래서 db file scattered read 대기가 발생한다.

인덱스가 없으면 조건을 줘도 full scan이 된다. hr.employees를 복사한 hr.emp(인덱스 없음)에 PK 같은 조건을 줘도 옵티마이저가 다른 길이 없어 TABLE ACCESS FULL을 택한다.

SYS@ora19c> create table hr.emp tablespace users as select * from hr.employees;
SYS@ora19c> set autot traceonly exp
SYS@ora19c> select * from hr.emp where employee_id = 100;

--------------------------------------------------------------
| Id  | Operation         | Name | Rows | Bytes | Cost (%CPU)|
--------------------------------------------------------------
|   0 | SELECT STATEMENT  |      |    1 |    69 |     3   (0)|
|*  1 |  TABLE ACCESS FULL| EMP  |    1 |    69 |     3   (0)|
--------------------------------------------------------------
   1 - filter("EMPLOYEE_ID"=100)
튜닝 포인트 — 단 한 행(employee_id=100)을 찾는데 테이블 전체를 HWM까지 읽고 있다. 여기에 employee_id 인덱스를 만들면 by index rowid로 한 행만 집어 비용이 급감한다. "한 행을 위해 full scan"이 바로 튜닝 대상의 전형이다.
05
rowid scan & 인덱스
소량을 콕 집는 길 — 그리고 인덱스를 못 타는 함정

5.1rowid scan — 주소로 한 행

rowid는 행의 물리 주소다. rowid를 직접 주면 옵티마이저는 인덱스도 안 거치고 그 블록으로 직행한다 — TABLE ACCESS BY USER ROWID(Cost 1). 인덱스를 경유해 얻은 rowid로 읽으면 TABLE ACCESS BY INDEX ROWID다. 둘 다 single block I/O라 db file sequential read가 발생한다.

형식구성
extended (v8, 18자)데이터오브젝트(6) + 파일#(3) + 블록#(6) + 행#(3) · 예 AAAS2nAAHAAAAD0AAD
restricted (v7)블록#(8) . 행#(4) . 파일#(4) · 예 000000F4.0003.0007

dbms_rowid.rowid_object/relative_fno/block_number/row_number로 rowid를 분해해 "이 행이 몇 번 파일·블록·행에 있나"를 볼 수 있다.

5.2인덱스 스캔의 종류

인덱스는 root → branch → leaf 트리다. range scan은 leaf에서 시작 값을 찾고, 옆 leaf로 이어가며 "하나 더 있나?"를 확인하는 one-plus-one 방식으로 끝을 판단한다.

스캔언제
INDEX RANGE SCAN범위·비유일. 같은 값이 더 있나 leaf를 이어 확인
INDEX UNIQUE SCAN유일 인덱스 + =일 때만. 범위(<=)면 RANGE로 떨어진다
INLIST ITERATORin (a,b,c) — 값 수만큼 인덱스 탐색 반복
INDEX FULL SCAN인덱스만으로 충당될 때(예: 인덱스 컬럼만 SELECT)
AND-EQUAL (index merge)두 단일 인덱스의 rowid 결과를 교집합(and_equal 힌트)

5.3★ 인덱스를 "안 타는" 경우

인덱스가 있어도 옵티마이저가 버리는 패턴들. 면접·시험 단골이다.

조건계획이유
to_number(employee_id)=100TABLE ACCESS FULL컬럼에 함수를 씌우면 인덱스 값과 달라져 못 탄다 → 함수 기반 인덱스 또는 컬럼 가공 회피
employee_id is nullFULLNULL은 일반 B-tree 인덱스에 저장되지 않는다
is not null + 인덱스 컬럼만 SELECTINDEX FULL SCAN인덱스만으로 충당되면 탄다
결합 인덱스 없이 last='King' and first='Steven'access(last) + filter(first)인덱스는 last_name만, first_name은 테이블에서 거른다
핵심 원칙 인덱스 컬럼을 가공(함수·연산·형변환)하면 그 순간 인덱스는 죽는다. 가공이 필요하면 컬럼이 아니라 상수 쪽을 바꾸거나, 함수 기반 인덱스를 따로 만든다. 그리고 filter로만 처리되는 핵심 조건이 보이면 인덱스(또는 결합 인덱스) 기회다.

5.4결합(조합) 인덱스 — 선두 컬럼이 전부

자주 함께 쓰는 컬럼은 하나의 결합 인덱스로 묶는다. (last_name, first_name)를 만들면 두 조건이 한 번의 access로 처리된다(앞의 filter 문제 해소). 단 선두 컬럼 선택이 성패를 가른다.

왜 선두 컬럼이 중요한가 — 전화번호부 결합 인덱스 (last_name, first_name)는 전화번호부처럼 성(last)으로 먼저 정렬하고, 같은 성 안에서만 이름(first)으로 정렬돼 있다. 그래서 성을 알면 한 구간으로 좁혀지지만, 성을 모르고 이름만 알면 그 이름이 책 전체에 흩어져 있어 정렬이 무용지물이다.

그래서 선두 컬럼은 이런 컬럼을 골라야 한다:

  • 혼자서도 자주 쓰이는 컬럼을 앞에 (선두가 조건에 없으면 위처럼 흩어져 못 좁힘)
  • 범위를 많이 줄여주는(선택도 높은) 컬럼을 앞에
  • = 비교로 쓰는 컬럼을 앞에 — 범위(BETWEEN·>) 조건을 선두에 두면, 그 구간 안에서 후행 컬럼은 연속 정렬이 아니라서 뒤 컬럼으로 더 못 좁힌다
direct path read 보너스 — full scan을 parallel로 돌리면 버퍼 캐시를 우회해 PGA로 직접 읽는다(direct path read). 이때 대상 테이블의 dirty buffer를 먼저 데이터파일에 내려쓰는 partial checkpoint가 일어나며 enq: TC - contention이 보일 수 있다 — 어제 배운 DBWR·checkpoint가 여기서 다시 등장한다.

5.5인덱스 스캔 종류 총정리 70일차

같은 인덱스라도 SQL 모양에 따라 스캔 방식이 갈린다. 실측으로 본 종류들을 정리한다.

스캔동작 · 언제
INDEX RANGE SCANleaf에서 시작값 찾고 옆으로 이어 읽기. 비유일·범위 조건
INDEX UNIQUE SCAN유일 인덱스 + = → 정확히 1행
INDEX SKIP SCAN결합 인덱스에서 선두 컬럼 조건이 없을 때, 선두값 종류마다 인덱스를 건너뛰며 탐색
INDEX FULL SCANroot→leaf 수직 진입 후 leaf를 순서대로 끝까지. 정렬된 결과라 ORDER BY를 흡수(sort 생략)
INDEX FAST FULL SCANleaf 블록을 multiblock으로 싹 읽기. 더 빠르지만 정렬 안 됨
INDEX FULL SCAN DESCENDING뒤에서 앞으로 → ORDER BY … DESC 흡수 (index_desc 힌트)
INDEX FULL SCAN (MIN/MAX)MIN()/MAX()는 인덱스 한쪽 끝만 보면 끝
INLIST ITERATORIN (a,b,c) — 값 수만큼 RANGE SCAN 반복

"이 SQL을 쓰면 → 이 plan이 나온다"로 다시 보면 (EMP_NAME_IDX = (last_name, first_name) 결합, EMP_IDX = employee_id 유일 기준):

SQL (조건/정렬)나오는 plan이유
where last_name='King'INDEX RANGE SCAN선두 컬럼을 =로 짚음 → 정석
where first_name='Steven'INDEX SKIP SCAN선두(last_name)가 없고 후행만 → 선두값마다 건너뛰며 탐색
where employee_id=100 (유일)INDEX UNIQUE SCAN유일 인덱스 + =
select last_name, first_name from empINDEX FULL SCAN읽을 컬럼이 전부 인덱스에 있음 → 테이블 안 감
… order by employee_idINDEX FULL SCAN인덱스가 이미 정렬돼 있어 sort 생략
… order by employee_id descINDEX FULL SCAN DESCENDING인덱스를 뒤에서 앞으로
select min(employee_id)INDEX FULL SCAN (MIN/MAX)한쪽 끝 한 블록만
select min(x), max(x) 함께일반 INDEX FULL SCAN ⚠한 스캔으로 양 끝을 못 집음 → 전체 읽음
where gender in ('f','m')INLIST ITERATOR값마다 RANGE SCAN 반복
SKIP SCAN이 유리한 조건 결합 인덱스 (gender, sal)에서 where sal between …처럼 선두(gender) 조건이 없을 때, 옵티마이저는 선두 컬럼의 고유값이 적고(예: gender = m/f 둘) 후행 컬럼의 고유값이 많을 때 SKIP SCAN을 택한다. 선두값(m, f)마다 서브 인덱스를 훑는 셈이라, 선두 고유값이 많으면 오히려 비효율이라 안 쓴다. 강제하려면 index_ss 힌트.
MIN/MAX 함정 — min(c)나 max(c) 단독은 INDEX FULL SCAN (MIN/MAX)로 양 끝만 본다. 하지만 select min(c), max(c)처럼 같이 쓰면 한 번의 스캔으로 양 끝을 못 집어 일반 INDEX FULL SCAN으로 전체를 읽는다. min … union all … max로 쪼개면 각각 (MIN/MAX)가 적용돼 빨라진다.

무엇으로 뽑고, 결과의 숫자를 어떻게 읽나

6.1도구 심화 — autotrace 모드 & display 포맷

도구 3종(autotrace · EXPLAIN PLAN · display_cursor)과 "예상 vs 실제(E-Rows/A-Rows)"는 §2.1 실행계획 읽기에서 다뤘다. 여기서는 70일차에 더 자세히 본 모드·포맷 옵션만 짚는다.

autotrace 모드 (set autot …) on(계획+결과+통계) · on exp(결과+계획) · on stat(결과+통계) · traceonly(결과 숨김, 계획+통계) · trace exp(계획만) · trace stat(통계만) · off. "결과를 볼까 / 계획만 / 통계만"의 조합이다. (autotrace는 PLUSTRACE role이 있어야 통계까지 나온다.)
display 포맷 — dbms_xplan.display(...,'basic'|'typical'|'outline'|'advanced'). basic은 Id·Operation·Name만, typical(기본)은 Rows·Cost·Predicate까지, outline은 옵티마이저가 쓴 힌트 묶음(계획 고정용), advanced는 Column Projection까지 전부 보여준다.

6.2Statistics 숫자 읽기

먼저 — 실행계획 표 자체의 컬럼(display_cursor … allstats last)이 숫자의 1차 출처다.

컬럼의미
Starts그 operation을 몇 번 반복했나 (NL 안쪽이면 바깥 행 수만큼)
E-Rows / A-Rows예상 행수 / 실제 반환 행수. 둘이 벌어지면 통계 의심
A-Time실제 수행 시간(누적)
Buffers그 operation의 논리 I/O 블록 수(누적). 물리 포함. 튜닝 1차 지표
Reads물리 I/O 블록 수(누적) — 디스크에서 실제로 읽음

그리고 autotrace의 Statistics 블록:

항목의미
consistent getsSELECT의 논리 I/O(읽기 일관성 모드 블록 읽기). 튜닝의 핵심 지표
db block getsDML의 current 모드 블록 읽기(논리 I/O)
physical reads물리 I/O — 디스크에서 실제로 읽은 블록
recursive calls오라클 내부 SQL 수. hard parse 시 증가(딕셔너리 조회 등)
redo sizeDML이 만든 redo 바이트
sorts (memory/disk)PGA에서 끝난 정렬 / temp로 흘러넘친 정렬. disk가 0이 아니면 PGA 부족 신호
두 출처는 같은 것을 다르게 부른다 — 실행계획 표의 Buffers ≈ Statistics의 consistent gets(+db block gets), Reads ≈ physical reads. 표는 operation별로, Statistics는 문장 전체로 본다는 차이.
튜닝은 보통 consistent gets를 본다 실행 시간은 캐시 상태·부하에 따라 출렁이지만 논리 I/O(consistent gets)는 그 SQL이 한 일의 양이라 안정적이다. 같은 결과를 더 적은 consistent gets로 내도록 바꾸는 게 튜닝의 본질. latch: cache buffers chains 경합도 결국 과도한 논리 I/O가 원인이라 같은 처방으로 줄어든다.

6.3버퍼 피닝 (Buffer Pinning)

한 줄 비유 — 책 45권을 꺼내는데, 여러 권이 같은 책장에 꽂혀 있다. 책장 하나를 꺼낼 때마다 제자리에 도로 꽂지 않고 옆에 둔 채(=핀) 계속 빼면, 책장까지 걸어가는 횟수가 확 준다. 여기서 "책장까지 걸어가기"가 블록 읽기(논리 I/O)다.

이 기법이 왜 필요한지를 문제 → 원인 → 해결 순으로 보자.

① 문제 — 인덱스 읽기는 사실 '작은 NL 루프'

Id 2 ↔ Id 1 의 반복 실행계획의 두 줄이 이렇게 주고받는다.
  1. Id 2 (INDEX RANGE SCAN): 인덱스 leaf에서 조건에 맞는 rowid 하나를 찾아 Id 1에 넘긴다.
  2. Id 1 (TABLE ACCESS BY INDEX ROWID): 그 rowid의 테이블 블록을 읽어 결과 한 행을 만든다.
  3. 다시 Id 2가 다음 rowid → Id 1이 또 읽고 … 행 수(45)만큼 반복.
순진하게 보면 행마다 테이블 블록을 한 번씩 읽으니 45번 + 인덱스 ≈ 46번이다.

② 알고보니 + ③ 그래서 핀 — 블록 단위로 보면

그런데 같은 부서원(department_id=50) 행들은 테이블에서 몇 개의 블록에 몰려 있다. Id 1이 한 번 읽은 블록을 바로 풀지 않고 핀으로 붙들어 두면, 다음 rowid가 같은 블록을 가리킬 때 다시 안 읽고 재사용한다.

실측 (department_id=50, 45행)

HR@ora19c> select /*+ gather_plan_statistics */ * from hr.emp where department_id = 50;
-----------------------------------------------------------------------------
| Id | Operation                          | Name         | A-Rows | Buffers |
-----------------------------------------------------------------------------
|  0 | SELECT STATEMENT                   |              |     45 |       9 |
|  1 |  TABLE ACCESS BY INDEX ROWID BATCHED| EMP          |     45 |       9 |
|* 2 |   INDEX RANGE SCAN                 | EMP_DEPT_IDX |     45 |       4 |  ← 핀으로 절약
-----------------------------------------------------------------------------
핀이 정확히 뭔가 — 무엇을·왜·언제 푸나
  • 무엇 — 핀(buffer pin)은 그 블록의 관리 카드(buffer header)에 꽂는 "지금 사용 중" 표시다. 데이터 본문이 아니라 블록마다 붙은 머리표에 기록된다.
  • 왜 절약되나 — 핀이 살아 있으면 그 블록은 메모리에 그대로 있어, 다음에 같은 블록을 쓸 때 cache buffers chains latch를 잡고 해시 체인을 뒤지는 과정을 통째로 건너뛴다. 즉 "주소로 다시 찾아가는" 게 아니라 "이미 손에 쥔" 것이다. 논리 I/O(Buffers)와 latch 점유가 함께 준다.
  • 언제 풀리나 — 다른 블록으로 넘어갈 때, 또는 그 operation/fetch 단위가 끝날 때 푼다. 그래서 같은 블록이 연속으로 와야 효과가 있고, 중간에 다른 블록이 끼면 풀려서 절약이 사라진다(= clustering이 나쁘면 효과 없음).
효과를 키우는 두 가지
  • BATCHED — rowid를 모아 블록별로 묶어 접근(TABLE ACCESS BY INDEX ROWID BATCHED). 같은 블록을 한 번에 처리해 피닝 효과를 키운다.
  • clustering factor — 같은 키의 행들이 테이블에서 가까이 모여 있을수록(클러스터링 팩터가 좋을수록) 연속 rowid가 같은 블록에 떨어져 피닝 효과가 커진다. 흩어져 있으면 매 행이 다른 블록이라 절약이 사라진다.
39일차 Execute에서 본 buffer pin이 여기서 튜닝 효과(논리 I/O 절감)로 다시 등장한 셈이다.

6.4클러스터링 팩터 — 같은 테이블에서 24배 차이 71일차

버퍼 피닝이 잘 먹히려면 같은 키의 행들이 테이블에서 가까운 블록에 모여 있어야 한다. 그 "모인 정도"를 수치로 나타낸 게 clustering factor다.

어떻게 계산되나 인덱스 leaf를 처음부터 끝까지 훑으면서, rowid가 가리키는 테이블 블록 번호가 직전과 바뀔 때마다 +1. 그래서 —
  • 값이 테이블 블록 수에 가까움 → 좋음 (행들이 블록별로 뭉쳐 있어 블록이 잘 안 바뀜)
  • 값이 테이블 행 수에 가까움 → 나쁨 (행마다 블록이 바뀜 = 흩어짐)

실측 — 같은 데이터, 같은 결과(71,332행)인데 인덱스만 다름 (테이블을 object_id 순으로 정렬해 생성):

인덱스컬럼clustering factor같은 쿼리 Buffers
C_TAB_IDXobject_id (정렬된 순서 ✓)1,389 (블록수 1,421에 근접)1,548
C_TAB_NAME_IDXobject_name (흩어짐 ✗)36,060 (행수 71,332 쪽)36,516
약 24배 차이 — 똑같은 71,332행을 세는데, 좋은 인덱스는 1,548번, 나쁜 인덱스는 36,516번 블록을 읽는다. 나쁜 쪽은 행마다 다른 블록이라 핀이 매번 풀려 재사용이 안 된다. 한 순서로만 정렬 가능이라는 함정도 그대로다 — object_id로 정렬해 만들었으니 object_id 인덱스만 좋고, object_name 인덱스는 나쁘다. 그래서 가장 자주·많이 범위 조회하는 컬럼 하나를 기준으로 정렬해 둔다(CTAS ... ORDER BY, dbms_redefinition 등).
latch와 연결 clustering이 좋으면 같은 블록 연속 → 핀 재사용 → cache buffers chains latch를 잡는 횟수가 줄어든다. 즉 clustering factor가 좋다 = 논리 I/O(Buffers)도, latch 점유도 함께 준다.

6.5SQL TRACE & tkprof 71일차

display_cursor가 "한 SQL의 실행계획+실측"이라면, SQL TRACE는 한 세션이 돌린 SQL들의 수행 통계를 파일로 남긴다. 그 파일을 사람이 읽기 좋게 포맷하는 도구가 tkprof다.

HR> alter session set sql_trace = true;        -- 추적 켜기
HR> select * from hr.emp where employee_id = 100;  -- 추적할 SQL 실행
HR> alter session set sql_trace = false;       -- 끄기
$ tkprof ora19c_ora_xxxx.trc out.txt sys=no    -- 포맷 (sys=no: 내부 SQL 제외)

tkprof 결과는 SQL마다 Parse · Execute · Fetch 세 줄로 통계를 보여준다.

call무슨 통계
Parse파싱·실행계획 생성
Execute커서 실행 단계 — DML은 여기를 본다
Fetch행을 실제로 가져옴 — SELECT는 여기를 본다
열의미
disk물리 블록 수 (= 디스크 읽기, pr)
query일관성 읽기용 논리 블록 = consistent gets — SELECT면 이걸 본다
currentcurrent 모드 논리 블록 = db block gets — DML이면 이걸 본다
rows처리한 행 수
한 줄로 외우기 SELECT → Fetch 줄의 query(consistent gets)·disk를 본다. DML → Execute 줄의 current(db block gets)·disk를 본다. (Row Source Operation 줄의 cr=논리 블록, pr=물리 블록, pw=쓰기도 같은 뜻.) display_cursor의 Buffers/Reads와 같은 것을 파일 관점에서 보는 셈이다.

PART 0
등장인물 — 프로세스와 메모리
무대를 먼저 깔아야 모든 동작이 제자리에 들어맞는다

SQL 처리를 이해하려면 "누가(프로세스)" "어디에(메모리)" 접근하는지를 먼저 알아야 합니다. 두 종류의 프로세스와 두 종류의 메모리가 무대의 전부예요.

프로세스 둘

  • User Process — SQL*Plus·애플리케이션 등 SQL을 작성하는 클라이언트 측 프로세스.
  • Server Process — 그 SQL을 실제로 처리하는 서버 측 프로세스. 전용 서버(dedicated) 구성이면 세션마다 1:1로 붙는다. 보충 SQL*Net(Oracle Net)이 둘 사이 통신을 담당하며, 이 통신 대기가 SQL*Net message from client다.

메모리 둘 — SGA vs PGA

  • SGA(System Global Area) — 모든 세션이 공유하는 메모리. Shared Pool(Library Cache + Dictionary Cache), Buffer Cache, Redo Log Buffer 등.
  • PGA(Program Global Area) — 세션 전용 메모리. 커서의 작업 영역·정렬·바인드 값이 여기 있다.
보충 · 커서는 두 부분으로 나뉜다 Shared SQL Area(파스 트리·실행계획)는 Library Cache에 두고 여러 세션이 공유한다. Private SQL Area(바인드 값·fetch 진행 상태)는 PGA에 두고 세션마다 따로 갖는다. 이 분리 덕분에 같은 SQL의 실행계획을 한 번만 만들어 모두가 재사용(soft parse)하면서도, 세션별로 다른 값을 동시에 처리할 수 있다.
보충 · Shared Pool과 PGA도 '작업 공간'이 있다 Buffer Cache만 깊게 보면 균형이 안 맞는다. 나머지 두 영역도 처리에 직접 쓰인다.
  • Shared Pool = Library Cache(공유 SQL·실행계획) + Dictionary(row) Cache(테이블·컬럼·권한 등 메타데이터) + Reserved Pool(큰 청크용 예약 ~5%). parse가 여기서 일어난다(Part 1). 메타데이터 변경 경합이 row cache lock, 큰 청크 부족이 ORA-04031.
  • PGA 작업 영역(work area) — ORDER BY·GROUP BY·DISTINCT는 sort area, HASH JOIN은 hash area에서 처리된다. PGA에서 다 못 담으면 temp 테이블스페이스로 흘러넘쳐(spill) direct path read/write temp가 발생한다. 그래서 정렬·조인 튜닝은 PGA(pg_aggregate_target) 이야기로 이어진다(→ SQL 튜닝편).
한 줄로: parse·메타데이터 = Shared Pool, 블록 캐싱 = Buffer Cache, 정렬·해시 작업 = PGA. 세 곳이 역할을 나눠 맡는다.

보충 · 왜 바인드 변수가 중요한가 리터럴을 매번 박으면(=100, =101 …) SQL 텍스트가 달라져 해시값이 매번 바뀌고, 그때마다 hard parse가 일어난다. hard parse는 옵티마이저 연산 + shared pool에 새 청크 할당(→ shared pool latch, 심하면 ORA-04031)이라 비싸다. :emp_id 같은 바인드 변수를 쓰면 텍스트가 고정돼 soft parse로 재사용된다. 운영 튜닝의 1순위 항목.

1.2hard parse는 내부에서 무슨 일을 하나 62

위 ④ 캐시검색에서 같은 커서를 못 찾으면(cache miss) hard parse가 시작된다. 단순히 "계획을 만든다"가 아니라 아래 단계를 차례로 거친다.

단계하는 일
1. 공유 가능성 확인해시는 같아도 완전히 동일한 SQL인지(텍스트·바인드 타입·옵티마이저 환경·참조 객체)를 비교. 다르면 같은 부모 아래 child cursor를 새로 만든다.
2. Query Transformation옵티마이저가 같은 결과를 내는 더 나은 형태로 SQL을 재작성한다. view merging, subquery unnesting, predicate pushing, OR 확장 등.
3. Optimization (CBO)통계(행 수·카디널리티·클러스터링 팩터)로 후보 계획들의 비용(Cost)을 추정·비교해 최저 비용을 선택. 액세스 경로·조인 방식·조인 순서를 정한다.
4. Row Source Generation고른 논리 계획을 실제 실행 가능한 반복자 트리(=실행계획)로 변환. 실행계획에 보이는 TABLE ACCESS FULL·NESTED LOOPS가 이 단계의 산물.
5. Library Cache 적재완성된 커서(파스 트리+실행계획)를 Shared SQL Area로 적재. 이때 shared pool 청크를 할당한다(→ latch/mutex, 부족하면 ORA-04031). 이후 같은 SQL은 soft parse로 재사용.
soft parse와 갈리는 지점 soft parse도 ①문법·②의미·③권한·④캐시검색은 똑같이 수행한다. 차이는 ④에서 기존 커서를 찾으면 2~4단계(변환·최적화·생성)를 통째로 건너뛰고 그 계획을 재사용한다는 것이다. 그래서 hard parse의 진짜 비용은 CBO 연산(CPU)과 적재 시 메모리 할당 직렬화(latch)다. 바인드 변수가 ④의 해시를 고정해 2~4단계를 안 타게 만드는 것이 parse 튜닝의 핵심이다.
parent vs child cursor — SQL 텍스트가 같으면 하나의 parent cursor(v$sqlarea)로 묶이고, 텍스트는 같지만 바인드 타입·환경이 달라 계획이 갈리면 그 아래 여러 child cursor(v$sql)로 나뉜다. child가 과도하게 많으면(high version count) 그 자체가 parse 부하·공유 실패의 신호다.

그럼 그 단계들이 진행되는 동안 라이브러리 캐시를 어떻게 잡고 푸는가. 도구는 셋이다 — latch/mutex(해시 체인 탐색, 찰나) · library cache lock(객체 핸들의 존재·권한 고정) · library cache pin(객체 본문(heap) 접근). lock은 이름표(핸들)에 대한 권한, pin은 본문에 대한 접근이라 역할이 다르다.

순서무엇을 잡나 / 왜
① 해시 버킷 진입SQL 해시 → 버킷 체인을 뒤지려 latch/mutex를 찰나만 잡고 놓는다. 같은 커서를 찾으면 soft, miss면 hard로.
② 참조 객체 보호테이블·뷰 등 핸들에 library cache lock S(파싱 중 그 객체가 drop·변경되지 않게 존재·정의 고정) + 정의를 읽는 동안 본문에 pin S. 그래서 파싱 중 그 객체에 DDL(X)이 들어오면 충돌해 막힌다.
③ 새 커서 적재변환·최적화·생성으로 만든 새 child cursor를 적재할 때 그 객체에 lock X + pin X(독점). 동시에 shared pool에서 청크를 할당하느라 shared pool latch를 잡는다(부족하면 ORA-04031).
④ 해제적재가 끝나면 새 커서의 X lock/pin을 풀고, 참조 객체의 S lock/pin도 푼다. 이제 공유 가능 → 다음 세션은 soft parse로 재사용.
lock과 pin을 왜 나누나 lock은 핸들(이름표)에 대한 권한이라 "이 객체가 파싱 내내 살아있고 정의가 안 바뀜"을 비교적 오래 보장한다. pin은 본문(heap)에 대한 접근이라 "지금 실제로 읽거나 쓰는 그 순간"만 잡는다. 그래서 정의를 lock S로 붙잡아 둔 채, 본문을 읽을 때만 pin S를 걸었다 푸는 식으로 따로 움직인다. 최신 버전은 latch·일부 pin을 더 가벼운 mutex로 대체해 경합을 줄였다(cursor: pin S, library cache: mutex X 대기).

1.3Shared Pool 보호 장치 — Library Cache lock/pin/mutex 62

라이브러리 캐시의 커서를 탐색·실행하는 동안 다른 세션이 그 커서를 갈아엎지 못하게 보호해야 한다. 이를 위해 library cache lock(객체 핸들 보호)·library cache pin(실제 실행 중 보호)을 shared 모드로 잡고, 최신 버전에선 library cache: mutex를 쓴다. 경합 시 library cache: mutex X 대기가 발생.

Execute — 읽기 63
실행계획을 따라 블록을 찾는다 — Buffer Cache

execute가 시작되면 library cache lock/pin을 shared로 잡아 커서를 보호하고, 실행계획의 연산(예: INDEX UNIQUE SCAN → TABLE ACCESS BY INDEX ROWID)을 따라 블록 I/O를 일으킵니다. 인덱스는 root → branch → leaf → 데이터 블록 순으로 내려가며, 매 단계가 "이 블록을 읽어라"는 요청이에요. 그 요청을 처리하는 메커니즘 전체가 Buffer Cache입니다.

3.1버퍼 캐시란, 그리고 왜 해시인가

Database Buffer Cache는 데이터파일에서 읽은 블록의 복사본을 담아 물리 I/O를 최소화하는 메모리 영역이다. v$buffer_pool 기준 DEFAULT 풀은 8KB 블록 버퍼 37,202개.

보충 · 왜 해시 테이블을 쓰는가 버퍼가 수만~수백만 개인데 "이 블록이 캐시에 있나?"를 매번 전부 훑으면 O(n)이라 느리다. 블록 주소를 해시해 버킷으로 바로 점프하면 평균 O(1)에 가깝게 찾는다. 그래서 "오라클이 메모리 관리에 쓰는 하나의 방법이 hash table"이라고 한 것. 스택·큐·링크드리스트·해시를 공부하라는 당부가 여기서 나온다.

3.2블록 찾기 — hash → bucket → chain → buffer header

요청 블록의 DBA(Data Block Address = 파일#·블록#) + 블록 클래스를 해시해 hash bucket을 찾는다. 한 버킷에는 같은 해시값을 가진 버퍼 헤더들이 체인으로 매달려 있고, 버퍼 헤더는 실제 버퍼 캐시 위치를 가리키는 메타정보다. 이 hash table 구조는 SGA에 있고, 각 체인은 cache buffers chains latch로 보호된다.

버퍼 헤더는 버퍼 캐시 바깥(SGA)에 있는 작은 메타 구조다. 블록 하나(=버퍼 하나)마다 정확히 하나씩 있고, 그 버퍼의 주소·상태·LRU/해시 체인 포인터, 그리고 잠금(핀) 상태를 담는다. 즉 버퍼 헤더는 "그 블록을 대표하는 표딱지"이고, 실제 데이터는 그 헤더가 가리키는 버퍼에 들어 있다.

3.8buffer header lock(pin) — latch와 무엇이 다른가

3.2에서 본 버퍼 헤더에 거는 잠금이 buffer header lock, 흔히 buffer pin이라 부른다(수업 노트의 "block lock"). 가장 많이 헷갈리는 게 cache buffers chains latch와의 차이인데, 보호 대상도 보유 기간도 다르다.

  • chains latch — 해시 체인에서 버퍼 헤더를 찾는 순간만 잡는 찰나의 잠금. 체인(자료구조)을 보호.
  • buffer header lock(pin) — 찾아낸 버퍼의 내용을 쓰는 동안 잡는 잠금. 버퍼(블록 데이터)를 보호.

그래서 순서가 이렇다 (Part 5 트랜잭션 step 3의 "block lock 획득 후 chains latch 해제"가 이것): ① chains latch 획득 → 버퍼 헤더를 찾음 → ② 헤더에 핀을 건다(읽기 shared / 쓰기 exclusive) → ③ chains latch 즉시 해제 → ④ 핀을 쥔 채 버퍼 내용 처리 → ⑤ 끝나면 핀 해제. latch는 찰나, 핀은 작업이 끝날 때까지 유지된다.

보충 · 보유 기간이 commit이 아니라 execute까지인 이유 buffer pin은 그 블록을 만지는 순간에만 필요한 메모리 구조 보호다. 그래서 같은 UPDATE 한 줄이라도, 핀은 블록을 바꾸는 동안만 잡혔다 풀리고(Part 5 노트: "active set 만들기 전=execute 종료 전까지 버퍼 헤더 lock 해제 안 함"), 행을 보호하는 TX lock은 commit까지 따로 남는다. "메모리 구조를 잠깐 지키는 핀"과 "논리적 데이터를 트랜잭션 내내 지키는 lock"은 완전히 다른 층위다.

4.1읽기 일관성 (Read Consistency)

오라클의 핵심 강점 하나 — 읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다. fetch 도중 어떤 블록이 "내 쿼리 시작 이후 다른 트랜잭션이 바꾼" 상태여도, 내 결과는 흔들리지 않는다.

  • 기준은 스냅샷 SCN(쿼리 시작 시점, §3.9). 읽으려는 블록의 SCN이 이보다 크면 "내 쿼리엔 너무 최신"이다.
  • 그러면 그 블록의 ITL→undo를 따라가 변경 전 모습으로 되돌린 CR 블록(Consistent Read)을 메모리에 새로 만들어 읽는다. 원본(current)은 그대로 두고 사본만 만든다.
  • 그래서 한 블록이 메모리에 current 1개 + CR 사본 여러 개로 공존할 수 있다. SCN은 변경의 시간 도장(단조 증가)으로 이 일관성의 기준점이다.
문장수준 vs 트랜잭션수준 기본은 문장수준 일관성 — 각 SQL이 "그 문장 시작 시점"으로 일관되게 본다. SET TRANSACTION ISOLATION LEVEL SERIALIZABLE이나 READ ONLY를 쓰면 트랜잭션수준 — 트랜잭션 시작 시점으로 트랜잭션 내 모든 쿼리가 같은 스냅샷을 본다.

대상 행의 lock byte를 세워 행 잠금. 다른 세션이 같은 행을 이미 잠갔으면 대기 enq: TX - row lock contention
change vector 생성 → redo log buffer 복사 65
PGA에서 change vector 3개 생성 — ①undo segment header ②undo block ③대상 블록. 이를 redo log buffer에 redo entry로 복사하며 redo copy → redo allocation latch를 순서대로 획득(부족 시 redo writing latch). 경합 시 latch: redo copylatch: redo allocationlatch: redo writing — 세부는 Part 5.5 참고.

보충 · ITL이 정확히 뭔가, 왜 2개? ITL은 블록 헤더 안의 작은 표로, "이 블록을 건드린 트랜잭션"마다 한 줄(entry)이 들어간다. 한 entry에는 txid, undo 위치(이전 값을 어디서 찾는지), 락 상태가 담긴다. 처음 만들 entry 수는 INITRANS(테이블 보통 1, 인덱스 2)로 정하고, 모자라면 블록 여유 공간에서 MAXTRANS까지 늘린다. 노트의 "테이블에 무조건 2개"는 환경에 따른 값이며, 핵심은 "동시 트랜잭션 수만큼 ITL이 필요하고, 부족하면 enq: TX - allocate ITL entry로 줄 선다"는 점.

락 유지 기간 — 시험·면접 단골

  • buffer header lock(block lock) — active set을 만들기 전, 즉 execute 종료 전까지 유지.
  • row level lock(transaction layer) — commit/rollback(트랜잭션 종료)까지 유지.
  • 오라클이 동시성을 보장해도 "그 순간"은 직렬이다. DML은 block lock을 exclusive로 잡는다.
보충 · "q가 들어간 wait event는 enqueue" enq:로 시작하는 대기(US, SQ, TX, TM…)는 enqueue — 큐(FIFO) 알고리즘으로 순서를 지키며 기다리는 잠금이다. 반면 latch는 순서 보장이 없는 아주 짧은 스핀 잠금. 이 둘을 구분하면 대기 이벤트 절반이 정리된다.
returning into — DML과 SELECT를 한 번의 블록 I/O로 처리하는 절. 예: update ... set ... returning salary into :v; 변경과 동시에 결과를 받아 블록을 두 번 안 읽는다. 튜닝의 핵심.
PART 5.5
Redo · LGWR · Commit 65
변경 기록을 먼저 디스크에 남겨야 commit이 끝난다

블록을 바꾸면 그 변경의 기록(redo)을 먼저 안전하게 디스크에 남기고, 그게 끝나야 commit이 완료된다. 이것이 Part 5 step 5·6의 속을 들여다본 내용이다.

4.5.1redo의 목적과 발생 작업

redo의 목적은 단 하나 — 데이터베이스 복구. DB에 적용된 모든 변경을 재구성·재실행할 정보가 redo entry다. 장애로 메모리가 날아가도 redo를 재생(roll forward)해 복구한다.

  • redo 발생 작업: DML(INSERT/UPDATE/DELETE) · DDL(CREATE/ALTER/DROP) · SELECT ... FOR UPDATE
  • 대비 정리: undo의 목적 = rollback + read consistency / redo의 목적 = 복구 면접 단골

4.5.2redo entry를 만드는 latch 순서

redo copy latch 획득
PGA의 change vector를 redo log buffer로 복사하는 전 과정 동안 보유. 경합 시 latch: redo copy
redo allocation latch 획득 → 공간 확보
redo log buffer에 redo entry를 넣을 공간을 확보. 경합 시 latch: redo allocation. 공간이 부족하면 redo writing latch로 LGWR에 기록을 요청하고, free 공간이 날 때까지 log buffer space. 로그 스위치 중이면 log file switch completion. 확보되면 allocation latch 해제.
redo entry 복사 완료 → redo copy latch 해제
change vector를 redo entry로 복사 완료 후 copy latch를 놓는다.

4.5.3LGWR는 언제 작동하나 — 5가지 트리거

  1. commit / rollback 시
  2. redo log buffer가 1/3 찼을 때
  3. redo가 1MB 쌓였을 때
  4. 마지막 LGWR 작동 후 3초 지났을 때
  5. DBWR 작동 직전 (WAL 보장)
보충 · 두 가지 원칙 (Write Ahead Log · Log Force at Commit) WAL — dirty buffer를 데이터파일에 쓰기 전에, 그 변경의 redo entry를 redo log file에 먼저 기록한다. 그래서 LGWR가 DBWR보다 앞선다.
Log Force at Commit — 트랜잭션과 관련된 모든 redo entry를 log file에 저장한 후에야 commit이 완료된다. 이 둘이 "데이터는 잃어도 redo로 복구 가능"을 보장한다.

4.5.4log file sync — 가장 흔한 대기

서버 프로세스가 commit/rollback 하면 LGWR가 redo를 log file에 기록(sync write, redo synch writes 증가)하는데, 서버는 그 write가 끝날 때까지 멈춰 기다린다. 이 대기가 log file sync. sync write 자체가 짧아 보통은 문제가 안 된다.

log file sync가 문제 되는 경우

  • commit/rollback이 너무 잦을 때
  • redo log file의 디스크 I/O가 느릴 때 (LGWR의 log file parallel write 대기가 높으면 의심)
  • redo가 불필요하게 많이 생길 때 (→ direct path + nologging 고려)
  • redo log buffer가 너무 클 때

PART 6
Lock & 동시성 66
여러 트랜잭션이 부딪힐 때 — Part 5의 TX가 충돌하면

Part 5–5.5가 하나의 트랜잭션 안에서 undo·redo·lock이 생기는 과정이었다면, 여기는 여러 트랜잭션이 같은 자원을 동시에 원할 때 어떻게 충돌하고 진단·해소되는지다. LOCK은 동시 변경을 막아 객체를 보호하는 동기화 장치이고, enqueue(대기 행렬)로 순서를 지킨다.

먼저 — lock은 '쓰기'끼리의 문제다 이 장의 충돌은 전부 쓰기-쓰기 사이의 일이다. 오라클에서 읽기는 lock을 잡지 않고, 쓰기도 읽기를 막지 않는다(§4.1 읽기 일관성). 그래서 SELECT는 아무리 많아도 서로·DML과 줄 서지 않는다 — 남이 바꾼 블록은 undo로 옛 모습(CR)을 만들어 볼 뿐이다. 즉 TX/TM lock으로 직렬화되는 건 변경(DML·DDL)뿐이고, 읽기 동시성은 lock이 아니라 MVCC(다중 버전)로 해결한다. 아래의 모든 블로킹 시나리오를 "쓰기끼리의 경합"으로 읽으면 된다.

PART 7
저장 구조 & 공간 관리 67
블록이 디스크에 어떻게 배치되고, 빈 공간을 어떻게 찾나

Part 0–6이 "한 블록을 읽고·바꾸고·잠그는" 이야기였다면, 여기는 그 블록들이 디스크에 어떻게 담기고, 동시 insert가 빈 공간을 두고 어떻게 경합하나다. 64에서 미뤄둔 block class(segment header·free list·bmb)와 buffer busy waits의 정체가 여기서 풀린다.

7.1저장 계층 — Block ⊂ Extent ⊂ Segment ⊂ Tablespace

  • Block — 데이터 저장·최소 I/O 단위. 2K/4K/8K/16K/32K (db_block_size, 기본 8K). 오라클 블록 1개 = OS 블록 1개 이상.
  • Extent — 연속된 블록 집합. segment가 tablespace에서 쓰는 공간 영역. 할당: create·extend·alter / 해제: drop·alter·truncate.
  • Segment — 테이블·인덱스 등 하나의 객체. 1개 이상 extent로 구성, 데이터 증가 시 extent를 더 받아 확장.
  • Tablespace — 논리 단위. 1개 이상의 Datafile(물리 파일)로 매핑.

freelists 1은 두 세션이 한 블록에 몰려 경합 폭증, freelists 2로 세션 수에 맞추면 급감. ASSM은 그 중간이지만 사람이 freelist를 안 맞춰도 자동이고 enq: HW까지 가장 낮다.

ASSM 블록 여유공간 6단계: ① full ② unformatted ③ 0~25% free ④ 25~50% ⑤ 50~75% ⑥ 75~100% free

보충 · 왜 ASSM이 확장성 있나 FLM의 freelist는 사실상 "하나의 줄"이라, 동시성이 높아지면 그 줄(과 segment header)에 직렬화 경합(buffer busy waits)이 생긴다. freelists 개수를 늘리면 완화되지만 "동시 세션 수에 맞춰 미리 수동 설정"해야 한다. ASSM은 블록별 비트맵을 여러 BMB(1st/2nd/3rd level)로 계층화해, 세션마다 다른 블록 후보를 자동으로 흩어주므로 수동 튜닝 없이 동시성에 따라 확장된다. 64에서 본 1st/2nd/3rd level bmb가 바로 이 계층 비트맵이다.
Q. 혼자(단일 세션) 쓰면 ASSM은 괜한 손해 아닌가? ASSM의 간판 이득은 동시성이라, 한 세션만 쓰면 그 이득은 사실상 0이 맞다. 하지만 손해도 0에 수렴한다 — 비트맵 한 번 확인 vs freelist에서 블록 꺼내기는 둘 다 메모리 안의 가벼운 북키핑이라 차이가 마이크로초 수준이고 디스크 I/O·redo 시간에 묻힌다. 게다가 FLM이 이기려면 freelists를 미래 동시성에 정확히 맞춰 수동 설정해야만 한다(위 표에서 freelists 2가 ASSM을 이긴 건 세션이 정확히 2였기 때문). 측정도 안 되는 시간을 아끼려고 수동 관리 부담 + 나중에 세션이 늘면 세그먼트를 재생성할 리스크를 사는 전형적인 조기 최적화(premature optimization)다.
Q. 그럼 ASSM만 있으면 되지 왜 FLM이 아직 있나? 시간 순서 때문이다 — FLM이 원래 방식이고 ASSM은 9i(2001)에 추가됐다. ① 하위 호환: 기존 DB엔 FLM 세그먼트가 가득해, 삭제하면 업그레이드 시 깨진다. ② 비상구: 초기 ASSM이 일부 워크로드에서 느린 사례가 있어 되돌릴 수단으로 남겼다. ③ 학습 대조군: freelists 1/2 vs ASSM을 비교해야 ASSM의 가치가 보인다(오늘 수업이 그 비교였다). 결국 FLM은 "구버전이라 못 지운 + 비상구 + 대조군"이고, 실무 기본·권장은 ASSM(auto). nologging·DMT처럼 더 나은 방식이 나와도 옛 방식이 호환성 때문에 남는 공통 패턴이다.

row migration — 실습으로 68

update로 행이 커졌는데 블록에 pctfree 여유가 없으면, 행이 새 블록으로 이사하고 원래 자리엔 "새 주소 포인터"만 남는다. 인덱스는 옛 rowid를 가리키므로, rowid scan이 포인터를 따라 한 번 더 점프한다.

  • chain_cnt는 analyze로만 채워진다 — dbms_stats.gather_table_stats로는 0으로 남고, analyze table ... compute statistics를 써야 migration된 행 수(334)가 채워진다. (시험 함정)
  • table fetch continued row(v$sysstat) — migration·chaining으로 "행을 한 번 더 따라간" 횟수. 인덱스 스캔을 돌리면 정확히 migration 수(334)만큼 증가 → 추가 I/O의 증거.
  • 해결 = 테이블 재구성: alter table ... move(행 재배치 → 단, 인덱스가 UNUSABLE로 바뀜) → alter index ... rebuild online(VALID 복구).
보충 · row migration vs row chaining row migration은 위처럼 update로 커진 행이 이사하는 것(pctfree로 예방). row chaining은 행 자체가 블록 하나보다 커서 처음부터 여러 블록에 걸치는 것(긴 행·LOB) — 이건 pctfree로 못 막고 더 큰 블록 크기가 답.

ITL 슬롯 부족 — enq: TX - allocate ITL entry 68

66에서 이름만 본 그 대기를 만드는 조건이 initrans(슬롯 수) + pctfree(슬롯 늘릴 공간)다. initrans 1 maxtrans 2 pctfree 0으로 만든 테이블에서, 한 블록에 든 여러 행을 3개 세션이 동시에 update하면 슬롯이 모자라 마지막 세션이 대기한다.

  • 진단 — v$session.blocking_session으로 막은 세션 추적(그 세션은 SQL*Net message from client=commit 안 함). 경합 객체는 v$segment_statistics의 'ITL waits'로.
  • v$lock 포인트 — allocate ITL entry는 TX REQUEST=4(Share)로 요청한다. row lock contention의 X(6)과 다르다.
  • 해결 — alter table ... initrans 2 pctfree 10. 단 이건 새 블록부터만 적용 → 기존 블록에도 먹이려면 alter table ... move(data_object_id가 바뀜) + rebuild online. row migration과 똑같은 처방(move+rebuild)이다.

latch

큐 알고리즘으로 순서를 지키는 잠금. 예: TX, US, SQ, TM.
buffer lock / pin
찾은 버퍼를 사용하는 동안 내용 보호(shared/exclusive). read by other session의 원인.
touch count
버퍼의 인기 점수. 3초 주기로 +1, ≥2면 hot 승격.

변경됐지만 아직 디스크에 안 쓴 버퍼. LRUW가 관리, DBWR가 기록.
undo
변경 전 값. rollback과 read consistency(CR)에 사용.
redo / change vector
변경 내역 기록. change vector들이 모여 redo entry가 되고 복구에 쓰인다.
ITL (Interested Transaction List)
블록 헤더 안, 블록을 건드린 트랜잭션 목록. txid·undo 위치·락 상태 보관.

변경의 단조 증가 시간 도장. 일관된 읽기의 기준점.
soft / hard parse
기존 커서 재사용(soft) vs 옵티마이저가 새 계획 생성(hard).
LGWR
redo log buffer의 redo entry를 redo log file에 기록하는 백그라운드 프로세스. commit/rollback·1/3·1MB·3초·DBWR 직전에 작동.
WAL (Write Ahead Log)
dirty buffer를 데이터파일에 쓰기 전에 redo를 log file에 먼저 기록하는 원칙. 그래서 LGWR가 DBWR보다 앞선다.
log force at commit
트랜잭션의 모든 redo entry를 log file에 저장한 후에야 commit이 완료되는 규칙.
log file sync
commit/rollback 후 서버 프로세스가 LGWR의 write 완료를 기다리는 대기 이벤트. 가장 흔함.
group commit
여러 commit을 LGWR가 묶어 한 번에 처리하는 것. 잦은 commit의 부담을 줄인다.
private redo strand
shared pool에 redo를 직접 쌓아 redo copy/allocation latch를 생략(zero copy latch)하는 10gR2+ 기능.
direct path / nologging
버퍼 캐시를 우회해 데이터파일에 직접 append(/*+ append */)하고, nologging으로 redo 생성을 최소화.
extent / segment / tablespace
연속 블록 집합(extent) → 객체 단위(segment) → 논리 저장 단위(tablespace, = 1+ datafile).
DMT / LMT
extent를 딕셔너리(fet$/uet$, enq: ST)로 vs 테이블스페이스 비트맵으로 관리. LMT가 표준.
FLM / ASSM
블록 여유공간을 freelist(수동)로 vs 비트맵(자동)으로 관리. ASSM이 buffer busy waits·enq: HW를 줄임.
pctfree / pctused
update 여유(10%, row migration 방지) / insert 재사용 기준(40%, FLM 전용).
HWM (High Water Mark)
세그먼트에서 데이터가 쓰인 경계선. full scan 범위를 정하며, 밀어 올릴 때 enq: HW 경합.
initrans / maxtrans
블록에 미리 만드는 ITL 슬롯 수 / 슬롯 전체 한도. 부족하면 enq: TX - allocate ITL entry.
table fetch continued row
row migration·chaining으로 행을 한 번 더 따라간 횟수(v$sysstat). chain_cnt는 analyze로만 수집.
full table scan / rowid scan
첫 블록~HWM를 multi-block로 통째 읽기 vs rowid로 한 행씩(by user / by index).
row migration / chaining
update로 커진 행이 이사(migration) / 행이 블록보다 커 여러 블록에 걸침(chaining).
TM lock
테이블(객체) 레벨 lock. DML 시 RX(3) 자동. v$lock의 ID1=OBJECT_ID.
TX lock
행(트랜잭션) 레벨 lock. 행을 잠그면 X(6). ID1=usn×65536+slot, ID2=slot seq#.
blocking_session
v$session에서 나를 막고 있는 세션의 SID. 거슬러 올라가면 보통 commit 안 한 세션.
deadlock
두 세션이 서로의 자원을 기다려 교착. 오라클이 자동 감지해 ORA-00060으로 한 문장만 취소.
